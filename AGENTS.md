# Repository Guidelines

## Operating Philosophy

`luosidingpm` 不是一个模板仓库，也不是一个只会生成文档的工具仓库。  
这里的每一个 skill，都对应产品经理在真实工作中的一种方法、一种判断路径、一种交付动作。

PRD、测试用例、A/B 实验方案、评审意见，这些都只是最终表现形式。真正重要的不是“把内容写出来”，而是：

- 是否先定义清楚问题
- 是否把问题拆解成可执行的模块
- 是否让上下游逻辑保持一致
- 是否让研发、设计、测试、运营都能继续工作
- 是否为后续实验、验证、迭代留下依据

因此，这个目录下的 skill 不能只追求“生成得快”，而必须优先追求：

- 问题是否成立
- 逻辑是否闭环
- 内容是否具体
- 结构是否完整
- 输出是否可交付
- 不同文档之间是否一致

如果一个 skill 只能帮助用户更快生成一份形式完整的文档，但不能帮助用户把问题想清楚、把方案讲明白、把风险暴露出来，那么这个 skill 仍然是不合格的。

## PM Working Rules

这个目录下的 skill 在工作时，默认必须体现产品经理的判断方式，而不是记录员式工作方式。

### 1. 先定义问题，再谈方案

任何方案型输出前，都必须先回答：
- 为什么要做
- 当前问题到底是什么
- 当前现状是什么
- 不做会怎样
- 做成后想改变什么

如果这些问题没有清楚，就不能直接进入页面设计、功能拆解、测试用例或实验设计。

### 2. 需求与方案必须区分

产品经理最常见的错误，不是不会写方案，而是把“想做的方案”误当成“已经成立的问题”。  
因此，任何需求都必须先分清：
- 用户当前说的是问题，还是方案
- 当前描述的是现状，还是目标
- 当前内容属于背景、模块、规则、异常，还是实现约束

如果这一步不清楚，后面的文档一定会漂。

### 3. 文档之间必须能相互支撑

好的 PRD 不是独立存在的，它必须能自然支撑：
- 测试用例怎么写
- 实验怎么设计
- 评审时怎么被挑战

如果一个需求写完后，研发看不懂做什么，测试看不懂怎么验，实验看不懂测什么，那就说明这个需求还没有真正写清楚。

### 4. 好的 PRD 不是写满，而是写透

什么叫写得好的 PRD，不是章节都有了，而是：
- 问题和目标一致
- 模块和目标一致
- 数据交互能支撑功能逻辑
- tracking 能验证目标是否真的达成
- 上下文前后没有冲突

也就是说，PRD 的价值不在字数和格式，而在于它是否让“问题 -> 方案 -> 系统 -> 验证”连成一条完整链路。

### 5. 产品经理必须有反向思维

一个成熟的产品经理不会只顺着需求写，还会主动问：
- 这真的是当前最重要的问题吗？
- 不做这个方案就一定无法解决吗？
- 有没有更低成本、更直接、更可验证的替代路径？
- 当前证据足够支持立项吗？

这个仓库里的 skill 也必须具备这种反向检验能力，而不是简单接受用户输入。

## Cross-Skill Consistency

这个目录下的 skill 不是彼此独立的输出器，而是同一套 PM 工作流的不同环节。  
因此，不同 skill 之间必须共享同一问题定义、同一目标、同一模块边界。

### 1. `prd-deep-interview`

作用：
- 通过深度对话生成一份 PRD
- 或对已有 PRD 做结构化评审

要求：
- 它是上游核心 skill
- 后续测试用例、A/B 实验、评审都应优先以这里定义的问题、目标、模块、规则为依据

### 2. `pm-review-board`

作用：
- 从多角色、多视角评审一份 PRD、方案或内容
- 重点是指出问题、挑战逻辑、发现缺口

要求：
- 它不是重写 PRD，而是审 PRD
- 它发现的问题，应优先回写到 PRD 进行修正，而不是在 review 中偷偷改写需求事实

### 3. `test-case`

作用：
- 根据需求或 PRD 生成、补全、评审测试用例

要求：
- 测试用例必须以 PRD 中的模块、规则、状态、异常和系统流程为基础
- 如果 PRD 没定义清楚，`test-case` 应指出缺失，而不是自行补全成事实

### 4. `ab-test-setup`

作用：
- 设计实验假设、指标、样本量、流量分配和实验结构

要求：
- 实验必须服务于 PRD 中定义的问题和目标
- 如果 PRD 目标是提升留存、转化、互动或效率，实验设计要围绕这些目标展开
- 不能脱离上游 PRD 单独创造一套问题定义

### 5. 下游 skill 发现上游文档缺口时必须指出

如果在 `pm-review-board`、`test-case`、`ab-test-setup` 中发现：
- PRD 有缺口
- 规则不清
- 模块边界模糊
- 数据交互不闭环
- 指标定义不足

必须明确指出这是上游问题。  
不能为了继续输出而把这些缺口偷偷补成既定事实。

## Repository Structure

`luosidingpm` 是一个产品经理技能集合目录。  
父目录用于聚合一组服务 PM 工作流的独立 skill，子目录各自负责一类明确任务。

当前和未来的 skill 可以包括但不限于：
- PRD 深度访谈与产出
- 测试用例生成与评审
- A/B 实验设计
- PRD 评审
- Figma 协作或设计交付相关能力

规则如下：
1. 每个子目录是一个独立 skill。
2. 每个 skill 负责一类明确工作，不要把所有内容都塞进一个 `SKILL.md`。
3. 共通工作原则写在父目录 `AGENTS.md` 中。
4. 每个 skill 的具体流程、模板、reference、scripts 由各自子目录维护。
5. 如果后续需要新增 skill，优先保持职责边界清晰，而不是继续把更多能力堆进已有 skill 中。
