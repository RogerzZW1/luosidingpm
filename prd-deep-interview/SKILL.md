# PRD 深度访谈技能

## 适用场景

本技能只能处理写一份 PRD 或评审一份 PRD 的需求，不能处理其他类型的需求。且不允许自己思考，必须严格按照skill.md来写作，且必须执行里面的每一个步骤和要求。

## 输出格式

最终的输出是要调用feishu CLI直接生成PRD，而不是在对话框里回复（详细参考底部飞书优先交付规则）如果用户没装feishu CLI告知用户，那就直接在对话框回复完整prd
所有的图表，excalidraw skills输出的、mermaid输出、以及如果后续调用别的skills生成ab实验以及test case的都必须和主PRD合并在一起，不允许分开文件。用竞品分析skills分析完后，内容也必须塞回prd正文中！（重要）

## 多轮深度对话协议

### 一、目标

通过多轮对话，让用户明确自己的需求。多轮深度对话的目的，不是把用户说的话搬进 PRD，而是通过连续追问、复述、校验和查漏补缺，确保 PRD 中每一个模块都达到“可直接写入、逻辑合理、上下文对齐、可交付”的标准。

### 二、适用范围

以下所有部分都必须通过多轮对话完成，不能只在业务背景阶段提问：

- 业务背景
- 当前现状与约束
- 竞品分析（如适用）
- 用户角色与使用场景
- 模块划分、模块详细描述
- 数据交互与系统流程
- 数据追踪设计
- 依赖、上线范围与灰度策略
- 存量兼容与迁移策略（如适用）

### 三、每一轮的固定输出

每一轮都必须输出：

- `当前理解`：复述当前已确认的信息，不允许直接跳到新问题
- `Skill.md追踪`：明确指出当前正在阅读skill.md第几行到第几行，模块标题是什么，【SKILL定位】SKILL.md L<start>-L<end>｜<当前模块标题>
- `缺口与风险`：指出还没说清、逻辑不闭环、上下文不一致或无法直接写入 PRD 的点
- `追问`：基于缺口继续提问，一个prd对应的模块可以问无数个问题，无数个轮次，但一个轮次就只问三个问题，这样可以让用户回答起来更方便，但是每个模块必须追问到满足当前模块完成标准为止，绝对不允许因为追问过一次，就默认该模块已经满足完成标准。



### 四、对话中的角色要求

在整个对话过程中，AI 不是记录员，而是具备产品判断能力的澄清者。必须：

- 最重要：你要判断用户说的内容是否真的满足当前章节的完成标准，而不是默认用户说了就算满足了，绝对不要用户说什么就填进去，用户的输入必须有你的判断（判断标准参考skill.md文中内容），如果不满足，你要明确指出来，并继续追问，直到满足为止（参考执行前置条件）！
- 重要：动发现逻辑冲突，而不是默认合理
- 以“能否写成清晰、可交付的 PRD”为标准推进对话

## 执行前置条件

在开始需求分析、追问或产出 PRD 之前，必须先完成以下动作。未完成任一动作，不得进入正式产出阶段。

1. 先完整读取本文件，从 frontmatter 到最后一个章节，一共有 [十三] 个章节，skill.md一共有接近1800行，不能只读前半段就开始工作，必须全部读完。读完后你首先告诉用户skill.md读了几行，第一部分整理清楚需求背景你会问什么样的问题，怎么样才算通过第一部分；以及如果需要竞品分析你会怎么做
2. 先列出本次任务必须覆盖的 PRD 模块，再进入逐段对话。至少要显式检查（且必须以这个顺序进行，不得跳过章节：
   - 业务背景
   - 当前现状与约束
   - 竞品分析（如适用）
   - 目标与非目标
   - 用户角色与使用场景
   - 功能模块与产品逻辑详细描述
   - 数据交互与系统流程
   - 数据追踪设计
   - 依赖、上线范围与灰度策略
   - 存量兼容与迁移策略（如适用）
   - 运营上线与GTM策略
3. 先识别当前步骤是否要求读取 reference 文件，未读取不得继续对应步骤。

在开始第一轮正式追问前，必须先向用户明确说明：本次 PRD 将按什么章节顺序推进，当前正在补哪一章，后续还需要补哪些章。如果没有先给出章节覆盖清单，不得直接进入功能性追问。

### 阻断条件

出现以下任一情况，必须停止正式产出，回到阅读、归类或追问阶段：

1. 只读了本文件前半段，尚未完成全文阅读。
2. 当前步骤需要 reference，但尚未读取对应文件。
3. 尚未列出本次任务必须覆盖的 PRD 模块。
4. 当前章节仍有明显缺失，却已经准备进入下一章节。
5. 仅基于常识或用户一句话，准备直接生成完整 PRD。

### 核心原则

1. 必须先对话沟通，再输出文档。
2. 多轮深度对话不是只发生在业务背景部分，而是贯穿背景、竞品、模块、交互、数据交互、埋点、存量兼容的每一个部分。
3. 每个模块都必须追问到“足以直接落入 PRD”的程度，不允许停留在抽象层。
4. 如果需求依附于现有产品或代码仓库，必须核查现状，不可只听口述。
5. 功能模块说明和产品逻辑说明合并输出，不拆成两个松散章节。
6. PRD 必须是文字加表格的结构化文档，不是口号式大纲。

### 当前章节只能问当前章节的问题

执行时必须遵守：

1. 当前章节未完成前，不得进入下一章节。
2. 当前章节只能追问当前章节需要的信息，不能越级提问后续方案细节。
3. 如果用户回答中混入了后续章节内容，要先归类，不要直接顺着展开。
4. 如果当前章节仍有缺失项，必须继续补齐，而不是先给一个“差不多”的版本。

### 越级提问的判定

以下行为视为越级提问：

1. 业务背景尚未补齐，就开始追问模块能力、交互路径、返回字段。
2. 当前现状尚未补齐，就开始追问 AI 是否默认首接、转人工怎么设计、知识库如何生效。
3. 目标与非目标尚未明确，就开始追问详细功能和状态设计。
4. 数据交互尚未明确，就直接输出埋点或灰度方案。

一旦发生越级提问，必须立即停止，回到当前章节继续澄清。

## PRD每一个章节的规范

无论当前在 PRD 的哪个部分，都只保留以下三类基础判断。  
这些规则的作用不是替代各章节自己的完成标准，而是作为所有章节都必须执行的最小判断动作。

#### 1. 判断用户输入是否满足当前模块要求

这一条要回答的问题是：**用户现在给出的信息，够不够支撑当前章节输出成结构化内容。**

判定动作：
- 检查当前输入是否已经覆盖了该章节所需的关键对象、关键关系和关键结果
- 检查当前输入能不能被写成该章节要求的格式，而不是只能写成一句概括
- 检查当前输入是否还缺少会直接影响章节成稿的关键信息

判断原则：
- 如果用户输入只能说明方向，不能说明对象、动作、规则、状态、结果，则不足以支撑输出
- 如果用户输入只能支持写一个标题，不能支持写正文、表格、规则或分支，则不足以支撑输出
- 如果某章节要求必须写字段、表格、状态、交互、指标、事件等结构化内容，而当前输入还不够填出这些内容，则不能进入正式成稿

通过标准：
- 当前输入已经足以支撑该章节写成结构化正文
- 当前输入已经足以填入该章节所要求的格式、字段或表格

不通过时的动作：
- 明确指出当前章节还缺什么
- 继续追问，直到具备最小可写条件
- 不允许因为用户已经回答过一次，就默认该章节已经可写

#### 2. 判断用户输入是否属于当前章节

这一条要回答的问题是：**用户刚刚说的内容，是否真的属于当前章节，而不是放错位置。**

判定动作：
- 先判断当前输入是在描述背景、现状、目标、方案、规则、异常、灰度、兼容还是别的内容
- 再判断这些内容是否和当前章节职责一致
- 如果不一致，要先纠正归类，再继续处理

判断原则：
- 在“业务背景”里不应直接写交互方案
- 在“当前现状”里不应直接写未来目标
- 在“目标分析”里不应直接写模块设计
- 在“数据追踪设计”里不应混入背景说明
- 在“功能模块”里不应混入需求来源或立项原因

通过标准：
- 当前输入与当前章节职责一致
- 当前章节中没有明显混入属于其他章节的内容

不通过时的动作：
- 直接指出“这段内容更适合放在哪一章”
- 先归类，再决定是否保留到当前章节
- 不允许把放错位置的内容原样收进当前章节

#### 3. 判断用户输入与上下文是否一致

这一条要回答的问题是：**当前输入和前面已经确认的内容，是否在逻辑上是一致的。**

判定动作：
- 检查当前输入是否与前文的问题定义一致
- 检查当前输入是否与前文目标一致
- 检查当前输入是否与前文角色、范围、约束一致
- 检查当前输入是否会导致后文章节逻辑断裂

判断原则：
- 背景中定义的问题，后面的方案必须真正在解决这个问题
- 目标中定义的指标，模块和流程必须能支撑
- 角色范围一旦确认，后面的功能不能无故切换使用对象
- 前文若定义为“旧能力迭代”，后文不能突然按“全新功能”来写

通过标准：
- 当前输入与前文没有明显冲突
- 当前输入能够承接上文，而不是另起一套逻辑

不通过时的动作：
- 明确指出冲突点在哪里
- 回退到冲突来源重新澄清
- 不允许带着冲突继续进入下一章节

### 五、对话中断恢复规则（必须执行）

在 `prd-deep-interview` 的正式流程中，如果出现以下任一情况，视为“技能中断”：

- 用户临时插入自由讨论
- AI 对某个点做额外解释后偏离了当前模块
- 双方连续多轮没有按固定协议输出
- AI 已经开始自然聊天式回复，未继续保持 PRD 模块化追问
- 当前章节还未完成，但对话已经偏离到其他模块、其他问题或泛讨论

一旦识别到中断，AI 不得直接继续随意往下写，也不得默认当前模块已经完成。  
必须先执行一次“协议恢复”，再继续当前模块的正式追问。


#### 触发这一规则时的目标

这条规则的目标不是“礼貌地把聊天拉回来”，而是确保：
- 对话恢复后仍然严格按 skill 的模块顺序推进
- 当前模块不会因为对话被打断而失控
- 用户始终清楚当前在 PRD 的哪个部分
- AI 始终清楚自己正在执行 `SKILL.md` 的哪一段规则

#### 协议恢复的固定格式

恢复时必须按以下顺序输出：

1. `当前理解`
- 复述当前模块中已经确认的信息
- 只总结当前模块，不允许直接跳到新问题
- 必须说明当前模块还在解决什么问题

2. `SKILL.md追踪`
- 明确指出当前正在执行 `SKILL.md` 的哪个模块
- 必须输出统一格式：
  - `【SKILL定位】SKILL.md L-L｜<当前模块标题>`
- 如果当前尚未完成该模块，也要明确写出“当前仍停留在该模块”

3. `缺口与风险`
- 指出当前模块里还没说清的点
- 指出逻辑不闭环、上下文不一致、信息不足、格式不足、无法直接写入 PRD 的地方
- 不能只写“还差一些细节”，要尽量指出是缺什么类型的信息

4. `追问`
- 基于缺口继续提问
- 每一轮最多问 3 个问题
- 但一个模块可以有无限轮追问
- 绝不允许因为已经追问过一次，就默认该模块已经满足完成标准
- 只有当当前模块满足本章通过标准后，才允许进入下一模块

#### 恢复后的执行要求

- 中断恢复后，AI 必须继续停留在当前模块，直到该模块通过
- 不允许借“已经问过几轮”为理由提前收口
- 不允许在当前模块未通过时进入下一章节
- 不允许一边恢复，一边偷偷开始输出后续章节内容
- 如果用户回答中再次混入后续章节内容，先归类记录，但当前轮仍然只处理当前模块



## PRD模块一：整理清楚需求背景

这一章不是简单收集“为什么要做”，而是要先判断：这个需求的业务问题是否真实成立、是否值得现在推进、是否已经足以进入后续方案设计。  
因此，这一章必须同时完成三件事：

1. 问清楚业务背景  
2. 判断用户输入质量  
3. 按固定格式输出结果  

如果任一项未完成，本章节不通过，不得进入下一章节。

### 硬规则清单（逐项打勾）（每轮对话都要带上）
- [ ] 需求来源（状态：未通过｜证据：）
- [ ] 当前问题（区分问题/解法）（状态：未通过｜证据：）
- [ ] 影响用户/流程/指标（至少1个指标）（状态：未通过｜证据：）
- [ ] 是否必须做（含不做后果&替代方案）（状态：未通过｜证据：）
- [ ] 为什么现在一定要做（状态：未通过｜证据：）
- [ ] 问题有效性判断（已有证据/仍缺证据）（状态：未通过｜证据：）
- [ ] 优先级判断（高/中/低+理由+缺证据）（状态：未通过｜证据：）

并且模块一“成稿输出”阶段（不是每轮追问）还要出现这些小节标题（脚本也能校验）：

1.1 需求来源 / 1.2 当前问题 / 1.3 影响范围与时机 / 1.4 问题有效性判断 / 1.5 优先级判断 / 1.6 反向思考结论

### 1.1 硬规则追问必须覆盖的问题

每一个必须问清的关键问题必须主动问出来，若用户主动说了，也要判断输入质量是否满足要求，如果不满足要求，也要继续追问，直到满足为止。

| 维度 | 必须问清的关键问题 |
| :--- | :--- |
| **1. 需求来源** | - 这个需求是谁提出的<br>- 来自哪个团队 / 哪个角色<br>- 是在哪个业务场景、项目节点、会议或反馈中提出的<br>- 为什么会进入当前优先级讨论 |
| **2. 当前问题** | - 当前真实问题是什么<br>- 这个问题发生在什么场景<br>- 为什么它会阻碍业务目标、用户目标或运营目标<br>- 用户当前说的是问题本身，还是他提出的一个解法 |
| **3. 影响用户与流程与指标** | - 哪类用户受影响最明显<br>- 是否是全量用户，还是某一类关键用户<br>- 不同用户类型受影响方式是否不同<br>- 当前受影响的是哪一段业务流程<br>- 问题发生在流程的哪个节点<br>- 是咨询、交易、履约、运营、审批、推荐还是其他链路受影响<br>- 当前问题影响哪些业务指标或过程指标<br>- 至少一个与问题直接相关的指标是什么<br>- 如果当前没有量化指标，必须明确说明“缺指标，待补充” |
| **6. 是否必须做这个需求** | - 如果不做会怎样<br>- 会带来什么损失、投诉、效率问题、收入影响或风险<br>- 不做是会持续恶化，还是只是维持现状<br>- 有没有更低成本、更直接、更可验证的替代方案 |
| **7. 为什么现在一定要做** | - 为什么这个问题必须现在推进<br>- 为什么不是以后再做<br>- 当前时机来自业务目标、项目窗口、市场竞争、资源条件，还是现有损失 |
| **8. 问题有效性判断** | - 当前问题是否有证据支撑<br>- 用户提出的方案，是“问题本身”，还是“他当前想到的一个解法”<br>- 是数据、案例、反馈、历史问题，还是经验判断<br>- 哪些证据已经有，哪些仍然缺失 |
| **9. 优先级判断（AI 完成）** | - AI 根据上面信息判断当前优先级是高 / 中 / 低<br>- AI 必须说明：<br>  - 为什么是这个优先级<br>  - 为什么不是更高优<br>  - 为什么不是更低优<br>  - 当前还缺哪些证据影响优先级判断 |

| 序号 | 维度 | 维度通过与不通过标准 | 通过例子及原因 | 不通过例子及原因 |

| 1 | 需求来源 | **通过标准**：已经明确说明是谁提出的、来自哪个团队或角色、发生在什么业务场景/会议/反馈中、为什么进入当前优先级讨论。<br>**不通过标准**：只知道“有人提了这个需求”，但不知道是谁、为什么提、在什么背景下提，也不知道为什么现在会被讨论。 | **例子**：`该需求由华南区销售主管提出，来源于 4 月招商复盘会。一线销售连续反馈，在线下谈判中商家需要“当场可兑现的利益凭证”，否则容易拖延签约，因此进入当前招商提效优先级。`<br>**原因**：已经明确了提出方、团队角色、提出场景和进入当前优先级的原因，需求来源成立。 | **例子**：`业务方想做一个客服机器人。`<br>**原因**：不知道是客服团队、运营团队、老板还是销售团队提出，也不知道是在什么业务背景下提出，无法判断需求来源是否真实有效。 |
| 2 | 当前问题 | **通过标准**：已经明确写清当前真实问题是什么、发生在哪个场景、为什么阻碍业务/用户/运营目标，并能区分“问题”与“解法”。<br>**进一步要求**：<br>- 必须同时包含“当前现象”和“真实问题”两层表达<br>- 必须说明问题发生在哪个具体业务节点<br>- 必须解释为什么该问题会阻碍目标结果<br>- 必须证明用户当前说的是问题而不是一个功能方案<br>**不通过标准**：用户只说“要做某个功能”或“要加某个入口”，但说不清真实问题；或者把解法直接当成问题。 | **例子**：`当前现象是夜间 18:00 后用户仍会发起咨询，但人工客服无人值守，导致高意向用户在咨询后直接流失。真实问题不是“没有 AI 机器人”，而是夜间场景下缺少即时承接与引导能力，导致用户无法继续完成报价或留资。`<br>**原因**：既说明了问题现象，也说明了问题本质；能看出“AI 机器人”只是候选解法，不是问题本身。 | **例子**：`我要做一个直播间定向发券功能。`<br>**原因**：这句话只说明了一个方案，没有说明当前问题是什么、为什么要发券、现在哪个环节出了问题。 |
| 3 | 影响用户与流程与指标 | **通过标准**：已经明确受影响的用户/角色、受影响的流程节点，以及至少一个与问题直接相关的指标；如果没有量化指标，也明确说明“当前缺指标，待补充”。<br>**不通过标准**：只说“影响体验”或“影响业务”，但没有说影响谁、影响哪段流程、影响什么指标。 | **例子**：`受影响最明显的是夜间访问车险比价页的未投保用户，他们在发起咨询后因无人承接而退出；受影响流程是“咨询 -> 报价 -> 留资”链路中的咨询承接节点；当前直接影响指标包括夜间留资率和 7 日复访率。`<br>**原因**：已经同时说明了受影响对象、受影响流程和受影响指标，影响范围完整。 | **例子**：`这个问题会影响整体业务表现。`<br>**原因**：没有说明是影响新用户、老用户、销售还是运营，也没有说明是咨询流程、交易流程还是运营流程，更没有指出任何指标。 |
| 4 | 是否必须做这个需求 | **通过标准**：已经说明如果不做会怎样，会造成什么损失/投诉/效率问题/收入影响/风险，并且当前需求被判断为解决该问题的较优路径之一，同时讨论过替代方案。<br>**进一步要求**：<br>- 必须回答“不做会怎样”<br>- 必须回答“问题会恶化还是维持现状”<br>- 必须回答“当前方案为什么比其他方案更值得优先推进”<br>- 必须讨论替代方案，并比较其优劣<br>- 不能只证明“做了更好”，而要证明“当前阶段值得做”<br>**不通过标准**：只能说明“做了更好”，但说不清不做会怎样；或者一问替代方案就答不出来。 | **例子**：`如果不做，销售仍然只能依赖线下口头承诺和慢审批流程，商家会在谈判窗口内流失，销售人效持续被拖低。替代方案包括继续邮件审批、外包客服或纯口头承诺，但这些方案都无法解决“现场即时交付与可信承诺”问题，因此当前方案在本阶段是更优路径。`<br>**原因**：已经说明了不做的后果，也比较了替代方案，并解释了为什么当前方案更适合当前阶段。 | **例子**：`这个功能做出来肯定更好。`<br>**原因**：没有说明不做会带来什么真实后果，也没有证明这是“必须做”而不是“做了更好”的优化项。 |
| 5 | 为什么现在一定要做 | **通过标准**：已经明确说明为什么必须现在推进，时机依据来自业务目标、市场竞争、资源窗口、项目节点、现有损失等至少一类。<br>**不通过标准**：只能说“最近想做”或“老板让做”，但说不清为什么不是以后做。 | **例子**：`当前 Q3 招商目标已经启动，华南区新增签约商家指标压力上升，同时晚间流量占比持续增加，现有承接能力已经影响线索转化，因此本需求需要在当前季度内先做试点验证，而不是延后到后续版本。`<br>**原因**：已经给出了明确的时机依据，包括业务节点和现有问题加剧的背景。 | **例子**：`最近团队都在聊这个，所以现在想做。`<br>**原因**：没有任何时机依据，无法判断当前是否真的必须现在推进。 |
| 6 | 问题有效性判断 | **通过标准**：已经说明当前问题有哪些证据支撑，证据来自数据、案例、用户反馈、历史问题或业务观察，并区分哪些证据已有、哪些仍缺失。<br>**进一步要求**：<br>- 必须说明至少一类证据来源<br>- 必须区分“事实证据”和“经验判断”<br>- 必须说明当前证据是否足以支撑立项<br>- 如果证据不足，也要明确“哪些还缺”<br>- 不能只用主观感觉支撑需求<br>**不通过标准**：只有感觉、经验判断或个别反馈，没有明确证据来源。 | **例子**：`当前已有 14 天夜间咨询基线数据，已知夜间发起聊天人数为 238，留资成功仅 21，留资率约 8.8%；同时已有高意向用户在夜间等待后离开的真实案例。当前证据已经足以证明夜间承接问题真实存在，但仍缺“留资成功到承保转化率”的补充证据。`<br>**原因**：同时提供了数据和案例，并明确哪些证据已有、哪些仍缺，问题有效性成立。 | **例子**：`我感觉用户会很需要这个功能。`<br>**原因**：只有主观判断，没有任何数据、案例、反馈或历史问题支撑，问题有效性不足。 |

# PRD模块二：当前现状与产品能力追问（整体不少于600字）

### 硬规则清单（逐项打勾）（每轮对话都要带上）
- [ ] 角色操作现状（角色/入口/怎么操作/能做什么不能做什么）（状态：未通过｜证据：）
- [ ] 系统现状（页面/配置入口/规则状态权限/数据来源/可复用能力/限制）（状态：未通过｜证据：）
- [ ] 当前能力差距（哪些已具备/哪些不能直接实现/哪些需新增改造兜底）（状态：未通过｜证据：）
- [ ] 当前业务数据现状（规模类/效率类/质量类等至少说明“有/没有/缺口”）（状态：未通过｜证据：）


### 这一章要解决什么问题

这一章不是为了直接写方案，而是为了回答：

1. 现在产品已经有什么能力，这次是在旧能力上新增、改造，还是替换
2. 现在不同角色分别怎么操作
3. 现在哪些链路已经存在、哪些链路缺失
4. 现在有哪些限制会影响新需求设计
5. 现在的业务数据表现如何

### 这一章的输出格式

必须形成以下结论，而且要从“角色如何操作”和“系统当前如何支持”两个视角同时写清：

#### 1. 角色操作现状
至少要说明：
- 当前涉及哪些关键角色
- 每个角色现在从哪里进入
- 每个角色现在具体怎么操作
- 每个角色当前能做什么、不能做什么
- 如果要做到目标需求，现在是否可以直接实现，还是还缺能力

#### 2. 系统现状
至少要说明：
- 当前有哪些页面、后台、配置入口、记录页、状态页
- 当前有哪些规则、状态、权限、数据来源
- 当前有哪些可复用能力
- 当前缺失哪些能力
- 当前有哪些明显限制会影响新方案

#### 3. 当前能力差距
必须明确回答：
- 按照目标需求，当前哪些环节已经具备
- 哪些环节不能直接实现
- 哪些需要新增、改造或人工兜底


#### 4 当前业务数据现状
至少要从以下几类中判断当前是否已有数据：
1. 规模类
- 当前业务量、使用量、发生量、处理量、访问量等规模数据
- 用来判断问题影响范围和优先级

2. 效率类
- 当前完成时长、响应时长、处理时长、等待时长、转化周期等
- 用来判断流程效率问题是否成立

3. 资源类
- 当前人力投入、系统投入、运营投入、成本消耗等
- 用来判断问题是否带来资源压力或成本问题

4. 结果类
- 当前转化、留存、成功率、完成率、投诉率、流失率等结果指标
- 用来判断问题是否已经影响业务结果

5. 结构类
- 当前问题主要集中在哪些时间段、渠道、页面、角色、场景、商品、任务类型
- 用来判断问题是否具有明显的分布特征


#### 5. 替代方案现状
在现状章节中，还必须补充当前团队已经存在或最容易想到的替代方案，而不是把替代方案放进竞品分析里。

至少要回答：
- 当前已经有人提出过哪些替代方案
- 这些替代方案是谁提出的（业务、运营、销售、研发、管理层等）
- 这些方案为什么被认为可以替代当前需求
- 各自的优点是什么
- 各自的缺点或不足是什么
- 为什么最终仍然需要进入当前方案讨论

写法要求：
- 不接受只写“还有别的方案”这种空话
- 必须写清替代方案提出方、成立理由、优劣势
- 替代方案属于现状判断的一部分，用于说明为什么当前方案值得继续推演

整体写法要求：
- 不接受只写“可复用部分 / 缺失部分”这种过度抽象总结
- 要尽量写清楚当前角色的实际操作方式，例如谁在哪里处理、通过什么入口处理、是线上还是线下处理、需要花多长时间、是否依赖人工协同
- 如果某项现状只是口头结论，也要显式标注“待核实”

如果用户提供了代码仓库或技术资料，可以在这一章把代码核查作为增强项补进去；但没有代码仓库时，这一章也必须完整成立。


### 必须追问的现状产品能力维度

无论是什么需求，至少要按以下维度把当前产品能力问清楚：

| 维度 | 要问什么 |
|------|----------|
| 入口 | 当前用户 / 商家 / 运营从哪里进入相关能力 |
| 角色 | 当前涉及哪些角色，各自能做什么 |
| 页面 | 当前有哪些前台页面、后台页面、配置页面 |
| 规则 | 当前有哪些业务规则、权限规则、状态规则 |
| 数据 | 当前数据从哪里来，谁维护，谁可见，谁可改 |
| 状态 | 当前是否有已回复 / 未回复 / 处理中 / 生效中等状态 |
| 复用 | 当前有哪些能力可以复用，哪些必须新建 |
| 约束 | 当前有哪些限制会影响新方案 |

### 抽象追问方法

现状追问必须从上一章已经确认的业务背景发散，而不是独立列一套固定问卷。也就是说，先看业务背景里已经明确了什么问题、目标、角色、场景和方案方向，再决定当前现状应该往哪里追问。

发散时，至少检查以下 5 类现状：

1. 核心场景现状
2. 关键角色现状
3. 相邻能力现状
4. 上下游系统现状
5. 当前缺口与限制现状

#### 1. 核心场景现状怎么识别

先从业务背景里的关键词和业务动作识别“需求发生在哪个场景”。

识别方法：
- 看背景里出现的场景词，例如：直播间、首页、搜索结果页、客服入口、结算页、运营后台、知识库上传等
- 看背景里出现的目标动作，例如：咨询、预订、发券、推荐、上传、管理、领取、审核等
- 看背景里出现的结果词，例如：留存、转化、互动率、回复率、首响、召回准确率等

把“场景词 + 动作词 + 结果词”组合起来，就能大致判断本次需求最核心的业务场景是什么。

这一类现状要问清：
- 当前这个场景是怎么跑的
- 当前这个场景中用户能做什么
- 当前流程里有没有已有机制支撑目标动作
- 当前流程里最大的断点或效率问题在哪里

#### 2. 关键角色现状怎么识别

先从背景里判断“谁受影响、谁发起、谁维护、谁承接”。

识别方法：
- 看背景中显式提到的对象，例如：用户、商家、运营、客服、知识库维护人员、审核人员
- 看目标是谁的目标，例如提升用户留存、降低客服压力、提升商家转化、提高运营效率
- 看这个需求要落地时，除了终端用户之外，是否还依赖商家端、运营后台端、人工处理端、中台角色等

这一类现状要问清：
- 每个角色现在分别从哪里进入
- 每个角色当前分别能做什么
- 哪个角色当前缺能力
- 哪个角色当前是人工处理，哪个角色当前有系统支持

#### 3. 相邻能力现状怎么识别

相邻能力指的是：和当前需求最接近、最可能被复用、最可能影响新需求设计的现有能力。

识别方法：
- 看用户提出的新需求，本质上依赖哪几个已有能力
- 看当前问题想解决的动作，在现有系统里有没有类似能力已经存在
- 看同一个业务链路中，是否已有“局部能力”可以复用，例如：
  - 已有直播间能力
  - 已有优惠券能力
  - 已有聊天能力
  - 已有知识库上传入口
  - 已有消息记录能力

这一类现状要问清：
- 这些能力现在在哪个页面或后台
- 当前支持哪些范围
- 哪些可以直接复用
- 哪些虽然存在，但规则或能力不足以支撑新需求

#### 4. 上下游系统现状怎么识别

上下游不是按技术架构死记，而是按“谁提供输入、谁承接结果、谁控制规则”来判断。

识别方法：
- 上游：给当前需求提供输入、状态、资格、上下文的系统或角色
- 中游：本次需求本身所在的核心产品模块或业务链路
- 下游：承接当前需求结果、记录状态、继续执行流程或做运营管理的系统或角色

判断标准：
- 谁提供必要输入，谁就是上游
- 谁消费结果或继续处理，谁就是下游
- 谁维护配置、控制规则、提供基础能力，也应纳入上下游现状追问范围

这一类现状要问清：
- 上游当前提供什么输入
- 下游当前如何承接结果
- 是否已有中台、后台、服务能力参与其中
- 哪些上下游能力会限制本次需求设计

#### 5. 当前缺口与限制现状怎么识别

在前四类现状问完后，最后要总结当前到底缺什么、卡在哪里。

识别方法：
- 对照目标，看当前场景中哪一步做不到
- 对照角色，看谁当前没有能力支撑
- 对照相邻能力，看哪些已有能力不能完全复用
- 对照上下游，看哪些系统当前没有打通或没有对应状态管理

这一类现状要问清：
- 当前已经有的能力是什么
- 当前完全缺失的能力是什么
- 当前有哪些限制会影响新方案
- 哪些限制是产品约束，哪些是系统约束，哪些是组织协同约束

在“当前现状与产品能力”阶段，除了问清当前入口、角色、页面、规则、状态和系统能力，还必须尽量追问当前业务的基线数据。  
如果没有现状数据基线，后续就无法判断问题规模、优先级，也无法在需求上线后评估效果。



### 以“直播间定向发券”为例，应该怎么问现状

不要直接问：
- 是否默认发券
- 优惠券发给谁
- 发券成功后用户怎么领

这些都属于后续方案。

这一章应该优先问：
- 当前直播间已有哪类互动玩法？
- 当前优惠券能力在哪些场景已经存在？
- 商家现在能不能主动对观众做定向动作？
- 用户当前在直播间能收到哪些实时权益或提醒？
- 运营当前如何配置活动、如何查看活动结果？
- 商家、用户、运营三方当前分别能看到什么、能做什么？


## PRD模块三：竞品分析怎么做

竞品分析不能只输出“产品定位、核心机制、优势劣势”这类总结性文本。对于需要写入 PRD 的竞品分析，必须尽量按“和我方 PRD 相同的拆解视角”去分析竞品，也就是：
在你阅读完这一部分的skill.md后，详细代码实操方法调用：- 优先使用 `agent-browser` 或 `playwright-interactive` 浏览器skills做页面访问、点击、导航和截图。若用户skills中含有类似的技能，则进行；若审查后没有，提醒用户安装


1. 这个产品解决什么问题
2. 它有哪些产品模块
3. 每个模块有哪些功能点
4. 页面长什么样、入口在哪里、交互怎么走
5. 是否存在明显的后端逻辑、推荐逻辑、召回逻辑、路由逻辑
6. 遇到异常、边界、失败场景时怎么处理

也就是说，竞品分析不是泛泛总结，而是要尽量为我方 PRD 提供可借鉴的“功能拆解、页面设计、交互设计、系统逻辑和产品判断依据”。

### 触发条件

以下情况必须进入竞品调研工作流：

- 用户要求做竞品分析或对标分析
- 用户提供竞品名称、网址、截图、录屏、调研笔记或关键词
- 当前需求属于明显已有成熟市场方案的功能，如推荐、问答、客服、预订、行程规划、搜索、内容消费等

### 是否需要竞品分析（必须主动询问）

在进入竞品分析前，必须主动询问用户：本次需求是否需要做竞品分析。
- 如果用户明确不需要，则跳过这一章，并标记“本轮未做竞品分析”
- 如果用户需要，则再进入调研输入确认和后续调研流程

### 调研输入要求

竞品分析开始前，优先确认以下输入：

1. 竞品名称
2. 竞品网址或截图
3. 想重点对比的功能范围
4. 是否已有用户调研笔记或现有结论

如果用户只给了一个网址或一个产品名，也可以开始，但必须先明确本轮调研聚焦的功能范围，例如：
- 旅游线路推荐
- 智能客服问答
- 门票预订对话入口
- 优惠券发放

### 能力拆解框架

竞品调研时，必须先用以下框架观察和记录，而不是直接总结。

| 维度 | 要观察什么 | 为什么重要 |
|------|------------|------------|
| 产品定位 | 这个功能在竞品里解决什么问题，服务哪类用户，处于什么业务场景 | 帮助判断竞品是在做问答、推荐、转化还是服务提效 |
| 模块拆解 | 功能可以拆成哪些大模块与功能点 | 让竞品分析可以直接映射到我方 PRD 的模块设计 |
| 页面与入口 | 页面长什么样，入口在哪里，在哪个链路中出现 | 帮助判断入口设计、触发时机和信息密度 |
| 页面元素 | 页面上有哪些关键元素：搜索、筛选、新增、删除、推荐位、状态标签、CTA、卡片、弹窗等 | 帮助还原真实交互，而不是只做概念总结 |
| 用户交互 | 用户如何操作，操作后会发生什么，状态怎么切换 | 帮助形成页面级交互借鉴 |
| 异常与边界 | 功能失败时、无数据时、条件不满足时、越界使用时怎么处理 | 帮助补齐我方 PRD 中最容易漏掉的异常逻辑 |
| 后端/数据逻辑推断 | 推荐逻辑、召回逻辑、路由逻辑、规则逻辑是否能从现象反推 | 对推荐、搜索、问答、Agent 类产品尤其重要 |
| 结果表达 | 系统最终返回给用户什么，结果以什么形式承载 | 帮助判断前台表达方式与下游承接方式 |

### 调研方式

竞品调研必须分两层进行：

#### 第一层：直接观察
通过以下方式直接收集证据：
- 打开官网或产品页面
- 访问具体功能入口
- 截图关键页面
- 记录入口、页面元素、流程和状态变化
- 阅读帮助中心、FAQ、文档、功能页

#### 第二层：设计场景反推逻辑
对于推荐、问答、Agent、预订、智能规划等产品，仅靠页面观察通常无法直接知道对方的后端逻辑。这时必须通过“设计 case”的方式反推对方逻辑。

也就是：
- 设计不同输入条件
- 构造不同意图
- 观察输出结果差异
- 再反推其背后的路由、召回、排序、规则或兜底逻辑

例如：
- 如果调研旅游线路推荐，要通过不同城市、不同偏好、不同时间条件去观察推荐结果差异，推测其召回和排序逻辑
- 如果调研客服或问答，要通过知识库内问题、知识库外问题、模糊问题、越权问题去观察其拒答、转人工、澄清、知识召回逻辑
- 如果调研商品推荐，要通过不同品类、不同行为历史、不同页面入口去观察推荐内容变化，推断其推荐策略

### 竞品分析输出结构

竞品分析最终必须同时输出两类内容：

#### 1. 产品层总结（文字）
至少包括：
- 产品定位
- 核心机制
- 优势
- 劣势
- 借鉴点
- 风险点

#### 2. 功能层拆解（表格）
必须按接近 PRD 的方式拆：

| 竞品 | 产品模块 | 功能点 | 页面/入口 | 页面元素与交互 | 异常与边界 | 证据来源 | 对我方借鉴 |
|------|----------|--------|-----------|----------------|------------|----------|------------|

#### 3. 后端/数据逻辑推断（如适用）
当需求涉及推荐、问答、RAG、Agent、预订、搜索等复杂能力时，必须补充一张“基于现象的逻辑推断表”：

| 竞品 | 场景 Case | 观察到的现象 | 可能的后端/数据/算法逻辑 | 可信度 | 备注 |
|------|-----------|--------------|----------------------------|--------|------|

写法要求：
- `场景 Case`：写你设计的测试场景
- `观察到的现象`：写页面、结果、分流或返回差异
- `可能的后端/数据/算法逻辑`：说明对方可能如何召回、排序、路由、过滤、拒答或兜底
- `可信度`：高 / 中 / 低

### 页面与交互分析要求

如果竞品页面或功能与我方需求高度相关，必须尽量按功能模块视角分析，而不是停留在高层总结。至少要补：
- 入口在哪里
- 页面元素有哪些
- 是否支持搜索、创建、删除、编辑、切换、筛选、导入、导出等关键动作
- 用户操作后有哪些状态变化
- 文案、提示、结果承载形式是什么
- 是否存在空态、失败态、拒答、fallback、人工兜底等处理

### 和工具的关系

- 页面访问、点击、截图等动作，应结合浏览器类 skill 或网页访问能力完成
- 若需要更结构化地沉淀系统流程图、架构图或模块流程图，在后续 PRD 正文阶段再调用 `drawio-diagram`
- 竞品分析本身先关注“观察、拆解、推断”，不在本章节直接生成正式系统图


## PRD模块四：目标分析

### 作用
本节不是写愿景口号，而是明确：
- 为什么要做
- 解决了前文哪个业务问题 / 用户痛点
- 上线后用什么判断是否成功
- 哪些代价不能接受
- 哪些范围本期明确不做

若本节内容无法对应前文的 `背景说明 / 现状问题 / 用户痛点 / 业务问题`，则视为不合格。

### 写法要求
- `业务目标`：必须对应前文业务问题，写清改善的是效率、成本、转化、风险中的哪一类；不能只写“提升体验”“优化能力”
- `用户价值目标`：必须对应前文用户痛点，写清用户完成什么任务会更快、更准、更省事；不能只写“提升用户体验”
- `核心指标`：必须能直接判断需求是否成功，优先结果指标；不能泛泛写“点击率”“使用率”除非本需求目标本来就是点击/使用
- `约束指标`：必须写清哪些指标不能恶化，通常包括投诉、误操作、失败率、性能、人工成本、风险指标
- `非目标`：必须明确写“不做什么”，用于收边界；不能写成“后续持续优化”

### 判断标准
- 每个目标都必须能回指前文至少一个问题
- 目标必须可验证、可复盘，不能是抽象口号
- 业务目标和用户价值目标不能混写
- 核心指标与约束指标不能重复
- 非目标必须能阻止需求范围继续外扩

### 错误示例
- 业务目标：提升业务体验
- 用户价值目标：让用户更方便
- 核心指标：使用率
- 约束指标：无
- 非目标：后续更多能力再补

### 正确示例（直播间定向发券为例子）
- `业务目标`：降低商家在直播中人工筛选和定向发券成本，提高营销动作执行效率，减少错发和漏发
- `用户价值目标`：让命中定向条件的观众更及时收到相关优惠，减少无关打扰，提升领券决策效率
- `核心指标`：定向发券功能使用率、目标用户领券率、商家完成一次发券的平均操作时长
- `约束指标`：错误发券率、非目标用户误领率、发券失败率、直播间主链路性能
- `非目标`：本期不支持站外触达、不支持实时动态重算定向人群、不扩展核销后的复购分析



## PRD模块五：用户角色与场景
这一部分不能只默认写“用户”一个角色，而是必须先判断本需求实际涉及哪些角色，再分别补充对应角色的使用场景。

### 1. 先识别角色，不要直接写结论

在进入“用户角色与使用场景”时，先判断当前需求涉及的是哪一类角色。至少检查：

- 终端用户
- 平台运营
- 后台管理员
- 客服/审核/人工处理人员
- 其他内部角色（如财务、风控、内容维护人员）

如果需求只涉及前台用户，不要硬加后台角色。  
如果需求涉及配置、维护、审核、人工兜底、内容更新、用户管理、数据查看等行为，则必须补充后台或内部角色。

### 2. 终端用户角色怎么写

如果存在终端用户，不能只写一句“平台用户”。要继续判断：

- 是否有不同类型的用户
- 新用户、老用户、已登录用户、未登录用户、潜客、已购买用户之间是否有差异
- 不同类型用户看到的入口、权限、结果是否不同
- 哪类用户是当前需求影响最大的重点对象

如果不同用户类型在流程、权限、页面展示或结果上有明显差异，必须拆开写，不要混成一个“用户”。

### 3. 后台运营/内部角色什么时候必须写

如果需求涉及以下任一情况，必须补充后台或内部角色的使用场景：

- 内容维护
- 知识库维护
- 配置管理
- 权限管理
- 用户管理
- 审核
- 数据查看
- 异常处理
- 人工接管
- 规则开关
- 灰度控制
- 运营干预

如果这些能力不存在于本次需求范围，也要明确说明“不涉及后台运营/内部角色”，而不是默认略过。

### 4. 每个角色都要写什么

无论是终端用户还是后台运营/内部角色，都至少要写清：

- 角色是谁
- 这个角色为什么会使用该功能
- 从哪里进入
- 在这个需求中会做什么
- 想达到什么结果
- 使用过程中最关键的动作是什么
- 是否有权限或边界限制

#### 5. 判断标准

这一部分只有同时满足以下条件，才算完成：

- 已识别当前需求涉及的全部关键角色
- 不同角色的目标和行为没有混写
- 对需要后台维护/配置/审核/人工处理的需求，已经补充后台或内部角色
- 如果不存在后台角色，也已明确说明“不适用”

如果当前只写了“用户”而没有判断是否存在后台运营、管理员或内部处理角色，则不能算完成。

#### 6. 输出格式
- 角色名称
- 角色适用场景详细描述：[为什么会用]+[][从哪里进入]+[会做什么]+[想达到什么结果]+[关键动作是什么]+[权限或边界限制是什么]



## PRD模块六：功能模块与产品逻辑详细描述

### 硬规则清单（逐项打勾）（每轮对话都要带上）
- [ ] 需求列表（状态：未通过｜证据：）
- [ ] 实现职责与依赖方评估（状态：未通过｜证据：）
- [ ] Excalidraw流程图+png图片（状态：未通过｜证据：）
- [ ] 大模块拆解表（状态：未通过｜证据：）

这一章不是在“把需求写成几个模块标题”，而是在回答一个更关键的问题：  
**前文已经确认的问题、目标、角色和场景，最终要落成哪些产品能力，这些能力分别由谁使用、在哪些页面发生、通过什么交互被完成。**
完整写法、详细拆法、正误示例、页面描述规则、用户交互详细描述规则、角色与职责评估等，见：- `references/module-design-method.md`（必须看）

因此，这一章的作用不是排版，而是做三件事：

1. 把前文的问题和目标，翻译成可交付的产品能力
2. 把前文的角色与场景，映射成可执行的模块和页面
3. 把这些模块继续拆到设计、研发、测试都能直接使用的粒度


### 6.1 本章的方法论定位

本章的核心不是先看页面，而是先看：

1. 哪些角色是本次需求的关键角色  
2. 每个角色在当前需求里要完成什么核心动作  
3. 为了支撑这些动作，产品必须提供哪些模块  
4. 这些模块再落到页面、状态、交互和系统逻辑  

也就是说，本章不是从“页面上要加什么东西”开始，而是从：

- 哪类角色有真实任务要完成
- 哪类角色只是被动结果接收方
- 哪类角色负责规则配置、人工接续、审核或治理
- 哪类角色需要系统支持、哪类角色只需路由或兜底

来反推模块。

#### 角色收敛规则

角色不是写完就结束，而是必须映射到模块。  
你可以把它理解成：

- 角色定义的是“谁在做什么”
- 模块定义的是“系统怎么承接这些动作”

如果这两者对不上，说明模块设计不成立。

1. 前文已定义的每个关键角色，都必须检查后文是否有对应模块承接  
2. 如果某个角色只需要被路由、被兜底、不需要完整能力，也必须在模块中显式说明  
3. 如果某个角色在前文被定义为本期核心角色，后文却没有对应模块，这一章不通过  
4. 如果某个模块服务多个角色，必须明确“主要服务角色”和“次要受影响角色”  
5. 如果某个角色本期不覆盖，必须显式写 `本期不覆盖`，不能默认省略  

### 6.2 本章必须输出什么

本章必须完成以下事情：

1. 先输出 `需求列表`
2. 再把需求拆成“大模块”
3. 每个大模块再拆成“小模块”
4. 每个小模块明确落到：
   - 哪个角色
   - 哪个页面 / 页面状态
   - 哪些关键交互
   - 哪些状态变化
5. 如有需要，再补 `实现职责与依赖方评估`

### 6.3 本章关键规则

#### 1. 先有需求列表
- 在进入详细模块描述前，必须先输出 `需求列表`
- 需求列表中的每一项都必须是：
  - 面向对象
  - 一句话产品需求描述
  - 可继续拆分
- 不能写成按钮、字段、弹窗、toast 这种控件级内容

#### 2. 大模块和小模块都必须是产品功能点
- 大模块：代表一类可独立评审的能力
- 小模块：代表大模块下可独立展开的功能点
- 两者都不能细到控件
- 两者都必须能和前文角色、目标、场景建立关系

#### 3. 页面描述只负责定义页面
- 页面描述只说明：
  - 这是哪个角色的页面
  - 这是哪个页面 / 哪个页面状态
  - 处于什么业务上下文
- 不在这一列写交互细节
- 页面只是承载，不是本章的核心

#### 4. 用户交互详细描述是核心输出
- 用户交互详细描述必须直接回答：
  - 用户或运营先看到什么
  - 在什么条件下可以做什么
  - 做了以后会发生什么
  - 页面如何变化
  - 成功和失败各自怎么表现
- 这一列必须比页面描述更细，而且必须能支撑：
  - 设计画页面
  - 研发写逻辑
  - 测试拆 case

#### 5. 模块必须和前文一一承接
- 如果前文已经确认了一个角色有核心动作，后面必须有对应模块
- 如果前文已经定义了一个关键目标，后面模块必须解释自己如何支撑这个目标
- 如果承接不上，就说明模块设计不成立

#### 6. 每个大模块结束后必须收口确认
- 总结本模块已收集信息
- 判断本模块是否已满足当前章节规则
- 明确告诉用户下一步进入哪个模块
- 用户确认后再继续

### 6.4 这一章怎么判断“写得好”

一个写得好的功能模块设计，不是“模块很多”，而是满足以下条件：

#### 1. 能承接前文问题
- 前文业务背景写的问题，在这里必须能找到对应模块
- 如果背景里定义的问题，在模块设计里没有被承接，说明模块拆错了

#### 2. 能承接前文目标
- 目标章节中定义的业务目标、用户价值目标，在这里必须能找到对应功能能力
- 如果模块与目标没有关系，只是“看起来想做”，说明模块无效

#### 3. 能承接前文角色
- 用户角色与使用场景里已经确认的关键角色，在这里必须被映射到模块
- 如果角色章节写了销售、主管、运营、商家，但模块里只写了销售动作，那说明模块设计不完整

#### 4. 能继续落到页面和交互
- 每个小模块都必须能继续展开到页面、状态、交互、规则
- 如果一个模块只能写一句概括，说明还不够落地

#### 5. 能支撑设计、研发、测试继续工作
- 设计能据此画页面
- 研发能据此理解功能边界
- 测试能据此拆出主流程、异常、边界
- 如果做不到，这一章就还没写透


### 6.6 本章不通过的典型表现

以下情况都说明本章还没写好：

- 模块只是把页面上看到的按钮和控件列出来
- 模块能看出“做什么”，但看不出“为什么做”
- 模块和前文角色、目标对不上
- 一个角色的核心动作在角色章节里写了，但在模块里没体现
- 页面描述写得很长，但真正交互和状态没写清
- 只有 happy path，没有条件分支、状态变化和失败结果
- 设计、研发、测试看完之后仍然不知道如何落地



## PRD模块七：数据交互与系统流程

这一章用于承接前文功能模块，说明需求背后的系统协同、后端规则、数据处理和算法逻辑。  
它不是简单重复页面交互，而是回答：

- 系统之间如何协同
- 后端如何处理规则和状态
- 数据或算法如何参与结果生成
- 失败和降级时系统如何处理

### 7.1 什么时候必须进入这一章

只要前文模块中出现以下任一情况，就必须进入这一章：

- 需要系统调用
- 需要状态变化
- 需要后端判断
- 需要数据处理
- 需要推荐 / 召回 / 排序 / RAG / 问答 / Agent
- 需要第三方服务、异步任务、回调、Webhook
- 需要明确状态由谁写入、由谁回传

如果只是一个纯页面微交互、文案调整、按钮展示规则，没有系统协同和数据逻辑，则可以不单独展开这一章。

### 7.2 进入这一章前的必读 reference

在开始本章正式追问或正式成稿前，必须先读取reference，未读取 reference，不得开始正式输出本章内容。reference: `references/system-flow-method.md`

### 7.3 本章需要输出什么

1. 逻辑文本描述部分：业务规则与处理逻辑
2. 系统交互/时序图

说明：
- 详细的文本拆解维度、Mermaid 规则、图类型选择、案例写法，全部见 `references/system-flow-method.md`
- 主文件这里不重复展开细节方法，只负责规定进入条件、必读 reference 和最终产出

### 7.4 本章通过标准

只有同时满足以下条件，本章才算通过：

- 已明确关键参与方
- 已明确系统交互与业务规则
- 已明确后端 / 数据 / 算法承接逻辑（如适用）
- 已明确失败与降级方案
- 满足条件时，已输出对应图表
- 图、表、文字之间没有互相冲突或重复堆砌

### 7.5 本章不通过的典型表现

以下情况都说明本章还没写好：

- 没有读取 `references/system-flow-method.md`
- 只写“前端调后端”
- 只写系统交互，不写规则和处理逻辑
- 只有图，没有文字关于后端逻辑的描述
- 图中缺关键参与方、参数、分支或状态源
- 前文模块和本章系统逻辑对不上


## PRD模块八：数据追踪设计怎么写

这一章不是让 AI 随便“列几个埋点”，而是要从 **需求背景、业务目标、产品模块、用户路径、系统流程** 里，一步一步拆出可落地的数据追踪方案。  
也就是说，先有目标，再有指标；先有流程，再有事件；先有业务含义，再有字段定义。

### 8.1 先从业务背景和目标倒推 tracking

数据追踪不能脱离需求背景单独设计。开始写这一章之前，必须先回到前文检查：

1. 这次需求的业务目标是什么
2. 这次需求想改善哪一个关键问题
3. 成功后最希望变化的结果是什么
4. 有哪些不能恶化的约束项

如果连“这次需求到底想改善什么”都没有说清楚，就不要开始定义埋点。

例如：
- 如果背景里写的是“18:00 后客服无人承接导致 7 天复访率下降”
- 那 tracking 的核心就不应该先写“打开聊天窗口”这种表层事件
- 而应该先想：
  - 晚间咨询用户是否被更快承接
  - AI 是否真正回答了问题
  - 有多少问题被转人工
  - 这些变化是否与复访率相关

### 8.2 拆解顺序：先目标，再流程，再事件

这一章必须按以下顺序拆解，而不是直接定义事件名：

1. 先定义业务结果指标
2. 再定义过程指标
3. 再从模块和用户路径里拆关键动作
4. 再把关键动作转成事件
5. 再给事件补属性、口径、触发条件和质量规则

### 8.3 怎么从文档内容里拆指标

#### 第一步：从“目标与非目标”里拆业务结果指标

要问：
- 这次需求最终想改善什么结果
- 这个结果如何被量化
- 哪个指标最能代表业务价值

例如：
- 目标是“提升 18:00 后发起咨询的登录用户 7 天复访率”
- 那这个就是业务结果指标

#### 第二步：从“产品逻辑”里拆过程指标

要问：
- 用户完成目标之前，会经过哪些关键过程
- 哪个过程成功，最可能推动业务结果改善
- 哪个过程失败，会导致整个目标失败

例如在 AI 客服里，过程指标可能是：
- AI 成功回复率
- 转人工率
- 拒答率
- 首次有效回复时长

也就是说，业务结果指标回答“最终结果有没有变好”，过程指标回答“为什么会变好或变差”。

### 8.4 怎么从模块里拆事件

事件不是拍脑袋定义出来的，而是从前文 `功能模块与产品逻辑详细描述` 和 `数据交互与系统流程` 里拆出来的。

拆事件时，先问：
1. 这个模块中，用户或系统做了哪些关键动作
2. 哪些动作会改变流程走向、状态结果或业务结果
3. 哪些动作值得被记录，用于后续分析或回溯

只有满足以下条件的动作，才适合定义为事件：
- 会改变流程状态
- 会影响业务结果
- 会影响用户体验
- 会影响后续系统行为
- 会影响后续分析判断

例如：
- `打开聊天窗口`
- `提交咨询问题`
- `AI 成功回复`
- `AI 拒答`
- `触发转人工`
- `知识文档上传提交`
- `知识文档处理成功`
这些都属于关键动作。

而像：
- 页面滚动
- 输入框聚焦
- hover 某个按钮
如果不影响关键业务结果，就不一定要在 PRD 里作为正式 tracking 事件定义。

### 8.5 事件命名怎么拆

事件名必须先有业务语义，再落成规范命名。

推荐拆法：
1. 先写中文业务动作
2. 再翻译成统一英文事件名
3. 再补触发条件和业务含义

例如：
- 中文动作：用户在首页聊天窗口提交问题
- 事件英文名：`chat_question_submitted`
- 触发条件：登录用户在首页聊天窗口输入问题并点击发送时触发
- 业务含义：一次新的咨询问题输入开始

要求：
- 事件名统一用英文 snake_case
- 不要直接把中文句子塞进“事件名”字段
- “谁在什么位置做了什么”要写在触发条件里，不要写成抽象短语

### 8.6 怎么拆属性

属性不是越多越好，而是围绕“后续分析要回答什么问题”来拆。

拆属性时，先问：
- 这个事件发生后，后续要按什么维度分析
- 分析时要区分什么对象、什么场景、什么结果

常见属性来源包括：
- 用户身份
- 会话上下文
- 页面或模块位置
- 问题类型
- 处理结果
- 文档或数据来源
- 失败原因

例如：
- 如果你后续想分析“哪些问题更容易转人工”
- 那么事件里就应该有：
  - `question_type`
  - `handoff_reason`
  - `session_id`

### 8.7 怎么拆口径与质量规则

口径不是附属说明，而是指标能不能被正确理解的前提。

至少要明确：
- 归因单位是什么
- 去重规则是什么
- 时间窗口是什么
- 哪些字段为空时不计入统计
- 延迟、空值、重复率怎么控制

否则就会出现：
- 同一个问题被重复统计
- 同一事件在不同系统里定义不同
- 最终指标失真

### 8.8 推荐输出结构

推荐按以下结构输出，不必再嵌套代码块：

- `数据追踪设计`
- `指标`：使用表格，字段为 `指标类型 / 指标名称 / 定义 / 公式 / 目的`
- `事件`：使用表格，字段为 `事件英文名 / 事件中文名 / 触发条件 / 业务含义 / 必填属性`
- `属性`：使用表格，字段为 `属性名 / 类型 / 是否必填 / 示例 / 定义`
- `口径与质量`：使用 bullet point，至少覆盖 `归因单位 / 去重规则 / 时间窗口 / 数据质量规则`

### 8.9 不合格写法

以下写法都不合格：
- 先列一堆事件名，再回头想业务目标
- 事件名只有中文，没有英文规范命名
- 触发条件写得过于抽象，例如“提交问题”“上传文档”
- 没有指标，只有事件
- 没有属性定义，只有字段名
- 没有口径、归因、去重和数据质量规则
- 埋点设计和前文目标、模块、系统流程脱节

## PRD模块九：依赖、约束与上线范围

这一节应写清：

- 上下游依赖
- 团队责任边界
- 上线前置条件
- 风险与开放问题

定义如下：

- `依赖`：这项需求要成功落地必须依赖的系统、接口、团队、第三方能力、配置条件
- `约束`：限制方案设计、开发方式、发布时间或范围的条件
- `上线范围`：本期到底覆盖谁、覆盖哪些模块、不覆盖哪些模块

推荐格式：

```md
## 依赖、约束与上线范围

| 类型 | 名称 | 说明 | 负责人 | 风险 |
|------|------|------|--------|------|
```

## PRD模块十：存量兼容与迁移策略

如果是迭代需求或配置变更，必须补这一节。至少回答：

- 是否影响历史数据
- 是否影响历史页面或历史实例
- 规则只对新建对象生效，还是对历史对象也生效
- 是否需要兼容层、版本字段、迁移脚本或人工处理
- 回滚如何处理

定义如下：

- `存量兼容`：旧数据、旧页面、旧实例、旧用户在新规则下如何继续可用
- `迁移策略`：如果不能直接兼容，如何把旧数据或旧配置迁移到新规则下
- `回滚策略`：上线后出问题如何恢复到旧逻辑

推荐格式：

```md
## 存量兼容与迁移策略

| 问题 | 结论 |
|------|------|
| 是否影响历史数据 | 是/否 |
| 是否影响历史实例 | 是/否 |
| 生效范围 | 仅新建 / 新旧都生效 / 灰度 |
| 兼容策略 | 兼容层 / 版本字段 / 保留旧逻辑 |
| 迁移策略 | 无 / 回填 / 脚本迁移 / 人工处理 |
| 回滚策略 | 如何回退 |
```

## PRD模块十一：灰度策略

任何高风险迭代、配置变更、支付链路改动，都必须判断要不要灰度。

必须回答：

- 是否需要灰度
- 如果不灰度，原因是什么
- 如果需要灰度，怎么控量
- 流量怎么分配
- 观察哪些指标
- 满足什么条件扩大流量
- 满足什么条件回滚

推荐格式：

```md
## 灰度策略

| 项目 | 内容 |
|------|------|
| 是否灰度 | 是/否 |
| 不灰度原因 | 如果为否，说明原因 |
| 灰度维度 | 用户 / 商户 / 区域 / 白名单 / 订单 |
| 流量策略 | 1% -> 10% -> 50% -> 100% |
| 观察指标 | 核心成功率、错误率、投诉率等 |
| 扩量条件 | 满足什么条件扩大流量 |
| 回滚条件 | 满足什么条件立即回滚 |
```
## PRD模块十二：上线后运营及GTM策略

这一章用于回答：需求上线之后，谁来推动它真正被使用、被传播、被验证、被放大。  
它不是产品功能本身的一部分，但如果缺少这一章，很多需求会出现“做出来了，但没人用、没人推、没人跟、没人复盘”的问题。

这一章的目标是把“上线”从一个技术动作，补充成一个业务落地动作。  
要明确：
- 什么时候上
- 先在哪些范围上
- 谁负责推广
- 用什么方式推动使用
- 上线后重点看哪些结果
- 如果效果不好怎么办

### 本章必须完成什么

在这一章中，至少要完成以下事情：

1. 明确上线时间与发布节奏
2. 明确首期覆盖范围和试点对象
3. 明确上线后的推广/运营动作
4. 明确上线后的观察指标和阶段目标
5. 明确责任人和协同角色
6. 明确效果不佳时的调整动作

---

### 本章关键规则

#### 1. 上线时间不能只写“待定”
上线时间至少要说明：
- 预计上线时间
- 是一次性上线、灰度上线、分批上线，还是试点上线
- 上线前依赖是否已满足
- 如果还未定时间，也要说明未定原因和预计决策时点

#### 2. 要明确“先给谁用”
不能只写“全量上线”或“面向所有用户”。

必须明确：
- 首期覆盖对象是谁
- 为什么先覆盖这些人
- 不覆盖的人暂时为什么不覆盖
- 是否需要分城市、分团队、分商家、分用户类型逐步推进

#### 3. 推广策略必须和角色匹配
GTM / 运营策略不是一套统一模板，而是要根据目标角色来写。

例如：
- 面向销售：培训、操作手册、激励机制、话术
- 面向商家：销售触达、招商话术、利益说明、入驻配合
- 面向用户：站内引导、消息提醒、活动说明、入口曝光
- 面向运营：SOP、数据看板、例行巡检、问题兜底

#### 4. 上线后要有观察期
不能上线就结束。必须明确：
- 上线后第 1 天看什么
- 第 1 周看什么
- 第 1 月看什么
- 哪些指标代表“有效采用”
- 哪些指标代表“需要调整”

#### 5. 必须能回答“如果效果不好怎么办”
至少要写：
- 是继续观察
- 还是补培训
- 还是调入口
- 还是调规则
- 还是暂停扩量
- 还是直接回退

## PRD模块十三：后续动作联动（必须主动问用户）

在你输出最终PRD之前，必须主动判断并询问用户是否需要继续进入后续动作，一共两个，是否要生成测试案例，是否需要A/B实验。  如果用户确认需要，则必须显式路由到对应 skill，而不是只口头建议。输出的test case以及ab-test-setup的内容，必须放回原始prd中，不要单独设立一个md文件

至少检查以下两类后续动作：

1. `test-case`
- 如果 PRD 已经明确了模块、状态、规则、异常和系统流程，主动询问用户是否需要生成测试用例
- 如果用户确认需要，则明确调用 `test-case` skill
- 传递给 `test-case` 的内容只应包括与测试直接相关的章节，例如：
  - 功能模块与产品逻辑详细描述
  - 数据交互与系统流程
  - 状态、规则、异常、边界
- 不应把业务背景、竞品分析、立项原因等无关章节整份传给 `test-case`
- 输出时必须明确告诉用户：接下来进入测试用例 skill，并说明本次传递的是哪些章节

2. `ab-test-setup`
- 如果需求存在明显的方案对比、策略实验、转化优化、体验验证、流量试点等特征，主动询问用户是否需要设计 A/B 实验
- 如果用户确认需要，则明确调用 `ab-test-setup` skill
- 传递给 `ab-test-setup` 的内容优先包括：
  - 业务目标
  - 用户价值目标
  - 核心指标
  - 约束指标
  - 本次想验证的关键方案点
- 输出时必须明确告诉用户：接下来进入 A/B 实验 skill，并说明本次传递的是哪些目标与指标信息

要求：
- AI 必须主动询问，而不是等用户自己提到 `test-case` 或 `A/B` 才进入下一步
- 如果判断不适合做测试用例或 A/B 实验，也要明确说明原因
- 一旦用户确认需要，必须进入对应 skill 的工作流，不允许停留在“建议你可以做”这一层


## 定稿前完整性检查

在输出正式 PRD 成稿前，必须先逐项检查以下内容是否完成。未完成时，不得宣称“PRD 已完成”，而必须明确列出缺失项并继续追问或标记待补充。

### 必查章节

1. 业务背景
2. 当前现状与约束
3. 竞品与替代方案（如适用）
4. 目标与非目标
5. 用户角色与使用场景
6. 功能模块与产品逻辑详细描述
7. 数据交互与系统流程
8. 数据追踪设计
9. 依赖、上线范围与灰度策略
10. 存量兼容与迁移策略（如适用）

### 必查动作

1. 检查是否还有缺失章节。
2. 检查是否还有只写了标题、没有正文的章节。
3. 检查是否存在前后冲突，例如背景写的问题与功能方案不一致。
4. 检查 tracking、灰度、兼容等章节是否被遗漏或只写了空结论。
5. 检查当前 PRD 是否已经达到“研发、设计、测试可以继续工作”的粒度。

### 输出规则

如果完整性检查未通过，只允许输出：

- 已完成章节
- 未完成章节
- 当前缺口
- 下一轮追问

### 需求澄清摘要（放在prd开头）

在正式输出 PRD 时，除了 PRD 正文，还应默认附带一份“需求澄清摘要”，方便用户对照最终文档与访谈过程。放在prd的最下方

这份摘要不是完整聊天记录，而是按模块压缩后的访谈纪要。

每个模块固定包含以下 4 项：
- `对话轮次`：该模块一共沟通了几轮
- `追问重点`：用一段话总结这个模块主要追问了哪些问题
- `用户回答摘要`：对用户原话进行压缩总结，尽量保留用户原话中的核心表达和关键词，不逐字照抄，也不完全改写成 AI 自己的话


写法要求：
- 不按“第 1 轮 / 第 2 轮”逐轮记录
- 不保留完整聊天 transcript
- 只按 PRD 章节模块输出摘要
- `追问重点` 和 `用户回答摘要` 各自写成一段自然语言，不再拆多条 bullet
- `AI 判断` 重点评价输入质量和章节完成度，而不是重复写需求内容
- 必须放在一个代码块中，段落与段落之间要有明显分隔符，例如 `---`

输出目的：
- 让用户可以快速核对：这一章问了什么、答了什么、为什么现在能进入下一章
- 让后续 review 时能回看需求是如何澄清出来的
- 保留需求演进脉络，但不污染正式 PRD 正文

### 飞书优先交付规则

对于个人日常使用，最终 PRD 默认以“可直接放入飞书文档”为第一交付目标。

规则如下：

1. 如果用户提供了文档标题，则直接以该标题作为 PRD 标题。
2. 如果用户没有提供文档标题，AI 应先根据需求生成一个明确、可直接用于飞书文档的标题。
3. 正式输出时，内容必须适合直接复制到飞书文档中使用，章节、表格、列表和代码块都要保持可读性。
4. 如果本次需求生成了 Mermaid 图，则 Mermaid 源码必须直接保留在 PRD 正文中，并使用代码块展示，方便用户复制到飞书文档的“文本绘图”组件中渲染。
5. 如果本次需求生成了 Excalidraw 图或系统框架图，则该图也必须作为 PRD 正文的一部分明确交付，不能只单独生成在对话外。至少要在 PRD 中说明：
   - 这张图的用途
   - 对应哪个章节
   - 用户需要如何导入、查看或继续编辑
6. 图不是附加品，而是 PRD 正文的正式组成部分。只要某一章节判断需要图，就必须把图或图源码一并交付给用户。

### 图表交付要求

- `Mermaid`：用于系统交互时序图、用户流程图、状态图，默认直接放入 PRD 正文的代码块中。
- `Excalidraw`：用于概念关系图、系统框架图、角色/部门协同图。若生成，则必须在 PRD 中明确标注该图对应的模块与用途。
- 当需求涉及多角色、多系统、多部门时，AI 应主动告知用户是否建议补一张 Excalidraw 图；无论调用与否，都要说明原因。
