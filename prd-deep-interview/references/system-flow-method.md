# 数据交互与系统流程方法

> 本文件用途：定义 PRD 中“数据交互与系统流程”的详细写法、文本拆解方式、Mermaid 时序图规则和输出格式。

## 七、数据交互与系统流程怎么写

这一章用于描述需求背后的系统协同、后端规则、数据处理和算法逻辑。  
它不是简单重复“前端调后端、后端返回结果”的过程说明，而是要把：**系统怎么交互、为什么这样交互、后端做了什么判断、数据如何处理、失败时怎么降级** 写清楚。

### 7.1 什么时候必须写这一章

只要需求不只是页面展示，而涉及系统调用、状态变化、后端判断、数据处理、推荐/召回/问答/Agent 等能力，就必须写这一章。

常见场景包括但不限于：

- 支付、订单、库存、优惠、积分、审批这类存在状态变化的需求
- 推荐、搜索、路线规划这类涉及召回、过滤、排序的需求
- 智能客服、知识问答、RAG、Agent 这类涉及理解、路由、检索、工具调用的需求
- 需要依赖第三方系统、内部服务、异步任务、回调、Webhook 的需求

反过来说，如果只是一个纯页面微交互、文案优化、按钮展示规则，没有系统协同、没有状态变化、没有数据逻辑，则可以不单独展开这一章。

和上文 PRD 的关系要明确：
- 如果在“功能模块与产品逻辑详细描述”里已经出现：
  - 需要查询用户数据
  - 需要校验资格
  - 需要调用知识库
  - 需要执行发券、扣减、预订、推荐
  - 需要人工兜底、回滚、状态切换
  那么这一章必须继续把这些逻辑从系统和数据层面补清楚。

例如：
- 在“直播间定向发券”需求里，前面模块已经写了“选择观众、选择优惠券、确认发放、结果反馈”，这里就必须继续写：
  - 观众列表从哪里查
  - 发券资格如何校验
  - 发券结果如何回写
  - 推送如何到达观众端
  - 失败时如何处理
- 在“智能客服”需求里，前面模块写了“AI 问答、转人工、知识库管理”，这里就必须继续写：
  - 问题如何分类
  - 知识库如何召回
  - 什么条件下转人工
  - 会话状态由谁维护

### 7.2 文本描述部分怎么写

这一章的文本部分仍然保留，但文本不再负责把整条交互顺序重复写一遍。  
文本和表格主要负责说明：**系统背后的规则、逻辑和处理方式**。

推荐用“文字 + 表格”的形式输出。

固定表格如下：

| 维度 | 内容 |
|------|------|
| 输入与依赖 | 用到了哪些输入、数据源、配置、上下文、权限、状态 |
| 判断与规则 | 有哪些校验、开关、互斥、叠加、优先级、资格判断 |
| 处理逻辑 | 后端做了哪些查询、召回、过滤、排序、生成、聚合、路由判断 |
| 输出与落库 | 返回给前端什么、写入数据库什么、触发下游什么 |
| 失败与降级 | 无结果、规则冲突、低置信度、超时、失败时怎么处理 |

写法要求：
- 这一部分要以业务 case 为单位写，不要写成抽象口号
- 每一项都尽量结合当前需求的具体角色和场景
- 用 bullet point 补充关键逻辑，不要只留表格

例如：

#### 营销优惠场景
- 输入与依赖：
  - 用户身份
  - 商品信息
  - 活动配置
  - 优惠券状态
  - 开关配置
- 判断与规则：
  - 哪些优惠可叠加
  - 哪些互斥
  - 谁优先级更高
  - 是否命中时间窗口
- 处理逻辑：
  - 先做资格校验
  - 再匹配活动
  - 再计算最终优惠
- 输出与落库：
  - 前端展示优惠结果
  - 后端记录命中活动与优惠明细
- 失败与降级：
  - 活动失效时如何提示
  - 配置冲突时如何兜底

#### 推荐/路线规划场景
- 输入与依赖：
  - 用户输入条件
  - 城市 / 时间 / 偏好
  - 候选 POI 数据源
- 判断与规则：
  - 哪些点位可召回
  - 哪些规则需要过滤
- 处理逻辑：
  - 召回
  - 过滤
  - 排序
  - 组合结果
- 输出与落库：
  - 返回路线结果
  - 是否保存草稿、推荐记录
- 失败与降级：
  - 无法生成路线时如何兜底
  - 数据不足时如何提示

#### 智能客服 / RAG / Agent 场景
- 输入与依赖：
  - 用户问题
  - 会话上下文
  - 用户身份
  - 知识库或工具能力
- 判断与规则：
  - 是否属于可答范围
  - 是否需要查个人数据
  - 是否触发禁答、拒答或转人工
- 处理逻辑：
  - 意图识别
  - 知识召回
  - 回答生成
  - 任务路由 / 工具调用
- 输出与落库：
  - 回复给用户的内容
  - 会话状态
  - 日志、命中记录、转人工记录
- 失败与降级：
  - 检索失败
  - 低置信度
  - 工具失败
  - fallback / 转人工

### 7.3 系统交互时序图（必须写）

系统交互图必须使用 `Mermaid`。  
对于系统协同、参数传递、同步异步、状态变化这类需求，默认输出 `Mermaid sequenceDiagram`，而不是普通流程图。

时序图中必须体现以下内容：

1. 参与方是谁
2. 谁发起、谁接收
3. 每一步关键参数是什么
4. 每一步的业务意图是什么
5. 返回结果有哪些类型
6. 哪些地方有前置判断
7. 哪些是同步、哪些是异步
8. 哪些地方有失败分支
9. 哪些地方会触发回调、重试、补偿、回滚（如适用）
10. 哪个系统是权威状态源
11. 前端如何感知状态变化

除了“谁调用谁”，图里还要体现**对象与对象之间的业务关系**。  
也就是说，不能只是技术链路，还要看得出：
- 为什么要调这个服务
- 这个服务返回的结果会影响什么
- 后续状态为什么会这样流转

例如在“直播间定向发券”里，图里不能只画：
- 商家端 -> 后端 -> 优惠券服务

还要体现：
- 为什么先查直播间在线观众
- 为什么要先做资格校验
- 为什么成功后要写记录
- 为什么还要通知观众端
- 什么情况下会失败
- 自动到账和手动领取为什么是两条分支

### 7.4 不同类型需求对应什么图

不同需求应使用不同图类型：

- **系统间调用、参数传递、同步异步、回调、状态回写**
  - 默认使用 `Mermaid sequenceDiagram`
- **用户流程、业务流程、页面流转**
  - 使用 `Mermaid flowchart`
- **状态机、状态切换**
  - 使用 `Mermaid stateDiagram-v2`

默认原则：
- 系统交互优先 `sequenceDiagram`
- 用户流程优先 `flowchart`
- 状态管理优先 `stateDiagram-v2`

如果一个需求同时存在三种表达需求，可以组合输出，但不要用一种图去强行表达所有内容。

### 7.5 Mermaid 输出规范

生成 `Mermaid sequenceDiagram` 时，至少遵守以下规则：

1. 参与方命名清晰
- 使用角色名或系统名
- 不要用模糊称呼
- 例如：
  - `actor Merchant`
  - `participant FE`
  - `participant BE`
  - `participant CouponSvc`

2. 调用必须写关键参数
- 不能只写“提交请求”
- 应写：
  - `提交发券请求(live_room_id, coupon_id, target_viewer_ids, operator_id)`

3. 返回必须写关键结果
- 例如：
  - `返回校验结果(pass/fail, reason_code)`
  - `返回发券结果(success_ids, failed_ids, fail_reasons)`

4. 分支必须用 `alt / else`
- 例如：
  - 校验通过 / 校验失败
  - 自动到账 / 手动领取
  - 可回答 / 不可回答 / 转人工

5. 图中只放系统交互逻辑
- 不要把大段解释性文字塞进图里
- 图负责顺序和分支
- 文字和表格负责规则说明

6. 生成 Mermaid 图后必须同时输出使用说明
- 要求在正式 PRD 中保留 Mermaid 代码块，使用 fenced code block 直接展示完整时序图源码，便于复制、预览和后续修改
- 要求在输出 PRD 时明确告诉用户：可以直接在飞书文档内通过「文本绘图」小组件粘贴 Mermaid 代码进行可视化和编辑
- 具体方式：打开飞书云文档，点击 `/` 或 `+`，选择「文本绘图」，将 Mermaid 代码粘贴到左侧代码区，右侧会自动渲染图表
- 说明：飞书的「文本绘图」原生支持 Mermaid 语法，可用于流程图、时序图、甘特图、类图等
- 注意：直接导入 Markdown 文件时不会自动识别 Mermaid，必须粘贴到飞书的「文本绘图」小组件内
- 如需批量导入 Markdown + Mermaid，可使用 `feishu-cli`：`feishu-cli doc import 你的文档.md --title "带Mermaid的文档"`
- 该方式可将 Markdown 中的 ```mermaid 代码块自动转为飞书画板，导入后继续编辑

### 7.6 推荐输出结构

推荐按以下结构输出：

- `数据交互与系统流程`
- `参与方`：用 bullet point 列出关键参与方
- `业务规则与处理逻辑`：使用表格，字段为 `输入与依赖 / 判断与规则 / 处理逻辑 / 输出与落库 / 失败与降级`
- `系统交互时序图`：使用 Mermaid `sequenceDiagram`
- `权威状态源与前端感知方式`：用 bullet point 补充说明

生成 Mermaid 图后，必须补充一句使用说明：
- 该 Mermaid 时序图源码应直接保留在 PRD 正文中，并使用代码块展示
- 用户可将该代码复制到飞书文档的「文本绘图」小组件内直接渲染和编辑
- 若需批量导入 Markdown 文档中的 Mermaid 图，可使用 `feishu-cli doc import 你的文档.md --title "带Mermaid的文档"`
