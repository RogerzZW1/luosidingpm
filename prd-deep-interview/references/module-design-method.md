# 功能模块与产品逻辑详细描述方法

> 本文件用途：定义 PRD 中“功能模块与产品逻辑详细描述”的详细写法、拆解方法、格式规范和示例。

## 六、功能模块与产品逻辑详细描述

这一部分必须使用结构化方式输出，不能继续采用“模块目标 + 模块概览 + 一段话 + 状态表”的旧写法。  
必须先输出 `需求列表`，再按“大模块 -> 小模块 -> 页面描述 -> 用户交互详细描述”的结构展开。  
最终目标不是写出一个好看的模块标题，而是让设计、研发、测试都能直接看懂每一个需求点在页面和交互上的真实表现。

### 6.1 需求列表

固定格式：

| 序号 | 角色 | 需求描述 |
|------|------|----------|

写法要求：
- 每一条需求都必须有明确角色
- 每一条需求都必须是一句话产品需求描述
- 不能是按钮级、字段级、toast 级描述
- 如果某个角色是前文定义的关键角色，但本期没有对应需求，必须明确写“不覆盖”
- 角色必须是上文 “用户角色“ 模块里面输出的用户角色内容，必须一一对应不得遗漏，一个用户角色可以对应多条需求列表的内容

### 6.2 大模块与小模块的关系、页面描述与用户交互详细描述的写法

以下规则统一放在这一节，不再拆成多个散小节。

| 序号 | 项目 | 定义 | 写法要求 |
|------|------|------|----------|
| 1 | 大模块 | 一个大需求点，代表一类可独立评审的产品能力，解决一类用户或业务问题 | 大模块必须能回答“它解决哪类问题”；可以覆盖多个页面、多个页面状态或多条操作链路；不能细到按钮、字段、toast |
| 2 | 小模块 | 大模块下可独立描述的功能点，是继续展开页面和交互的基本单元 | 小模块必须能落到具体页面、具体场景或具体状态；不能只是一个控件名称 |
| 3 | 页面描述 | 说明该小模块发生在哪个页面、哪个页面状态或哪个业务上下文中 | 页面描述只写“这是什么页面 / 什么状态页面 / 在什么业务上下文中”；不要把交互细节塞进这一列 |
| 4 | 用户交互详细描述 | 详细描述该小模块中用户/商家/运营如何操作、看到什么、在不同条件下会发生什么 | 必须使用分层 bullet；必须写清入口、动作、条件、状态、分支结果、失败反馈；不能写成一整段连续文本 |
| 5 | 角色要求 | 需求描述时必须明确当前能力属于用户端、商家端、运营后台端或多角色协同 | 需求模块本身要体现“这是哪一端的能力”；不能把商家端能力写成用户端能力，也不能漏掉后台端 |
| 6 | 状态要求 | 一个小模块可能对应多个页面状态，必须显式说明状态变化 | 至少写清从什么状态进入、做了什么动作、变成什么状态；不同状态下看到什么、能做什么 |

#### 1. 需求模块

写法要求：
- 需求模块默认指“小模块”
- 小模块必须是从“大模块”进一步拆出来的功能点
- 小模块不是控件，不是字段，不是 toast，不是局部 UI 元素
- 小模块必须能独立回答：
  - 这个功能点解决什么问题
  - 由哪个角色使用
  - 在哪个页面或页面状态中发生
  - 后续能否继续展开交互、状态、条件分支和结果

##### 如何从大模块拆成小模块

拆分原则必须尽量满足 MECE，即：
- **Mutually Exclusive（相互独立）**
  - 每个小模块关注的是不同的功能点，避免重复描述
- **Collectively Exhaustive（合起来完整）**
  - 所有小模块加起来，能够覆盖这个大模块的主要能力，不遗漏关键链路

拆分时按以下顺序思考：
1. 先明确大模块解决的核心问题
2. 再判断这个大模块中，用户或运营会经历哪些关键动作阶段
3. 每个关键动作阶段，单独拆成一个小模块
4. 检查这些小模块之间是否有明显重叠
5. 检查这些小模块加起来，是否已经覆盖了大模块的主流程、关键状态和主要结果

##### 用“直播间定向发券”举例

大模块：
- 直播间定向发券

这个大模块解决的问题：
- 商家在直播过程中，需要针对特定观众做促活和转化
- 因此需要在现有直播间和现有优惠券能力基础上，新增“向指定观众发券”的能力

拆分思路：
- 商家先要选人
- 然后选券
- 然后确认发放
- 发放后需要看到结果
- 被命中的观众需要接收并领取优惠券
- 商家或运营后续还需要查记录

正确示例：
- 商家选择发券对象
- 商家选择优惠券并确认发放
- 商家查看发放结果反馈
- 直播观众接收与领取优惠券
- 商家后台查看定向发券记录

为什么这样拆是对的：
- 每一项对应一个独立功能阶段
- 每一项都能落到具体页面或页面状态
- 每一项都能继续展开交互、状态、条件分支和结果
- 几项加起来，基本覆盖了“直播间定向发券”这个大模块的完整能力链路

错误示例：
- 点击发券按钮
- 搜索观众
- 勾选观众
- 选择优惠券下拉框
- 确认发放按钮
- 发放成功 toast

为什么错：
- 这些是控件、局部动作或反馈，不是需求模块
- 拆得太碎，失去产品功能层级
- 无法让研发、设计、测试按功能点评审
- 也无法自然组织页面、状态和交互逻辑

另一种错误示例：
- 直播间发券
- 直播间发券流程
- 直播间发券管理

为什么也不够好：
- 这些模块之间边界模糊
- 有明显重叠
- 无法看出每个模块分别承担什么能力
- 不满足相互独立

##### 判断一个小模块是否合格

一个小模块至少要同时满足以下条件：
1. 是一个功能点，不是一个控件
2. 可以明确对应角色
3. 可以明确对应页面或页面状态
4. 可以继续写交互细节
5. 可以继续写状态变化
6. 可以继续写成功/失败/边界场景

如果不能满足以上条件，就不应该定义为小模块。

#### 2. 页面描述

页面描述只负责说明：
- 这是哪个角色的页面
- 这是哪个页面或哪个页面状态
- 页面处于什么业务上下文

写法要求：
- 不在这一列写交互细节
- 不在这一列写条件分支
- 不在这一列写状态变化细节
- 重点是让人知道“这是哪个页面 / 哪个状态页面”

例如：
- 商家直播间控制台发券面板（对象选择态）
- 商家直播间控制台发券结果态
- 用户端直播间页面（领券提示态）
- 商家后台发券记录页

#### 3. 用户交互详细描述

这一列是整张表的核心，必须比“页面描述”更细，而且要接近你截图里的需求描述风格。

##### 格式整体要求
- 必须使用多级 bullet，不能在一行里用 `\n` 拼接
- 一级 bullet 表示一个主要交互阶段
- 二级 bullet 表示该阶段下的：
  - 展示内容
  - 操作
  - 校验
  - 成功结果
  - 失败结果
- 如需更多分支，可继续用三级 bullet
- 不要把“展示 / 操作 / 校验 / 成功 / 失败”写成一整行
- 每一项单独成行，保证复制到飞书后自动换行

##### 内容整体要求

“用户交互详细描述”不是对页面做一句话说明，而是要把某个需求模块在真实使用中的交互过程、条件分支、状态差异、失败处理写清楚。  
这一部分的目标是：让设计、研发、测试在不额外追问的情况下，能理解这个功能点在页面上如何发生、在不同条件下如何变化、在异常情况下如何处理。

在生成“用户交互详细描述”前，必须先按统一框架完成需求拆解，再把结果自然融入 bullet 描述中，不要求显式输出方法标签。

##### 一、统一拆解框架

每个需求模块在写详细交互前，至少要先从以下 6 个维度思考：

1. 角色与对象
- 谁在使用这个功能
- 是否涉及多个角色
- 角色之间是否有权限或可见性差异
- 被操作对象是什么，例如订单、优惠券、线路、会话、文档、用户

2. 页面与状态
- 这个功能发生在哪个页面或页面状态中
- 进入该状态前用户处于什么位置
- 页面当前承担的任务是什么
- 页面有哪些关键状态切换

3. 触发与动作
- 用户或运营从哪里开始操作
- 具体做了什么动作
- 动作是点击、输入、选择、拖拽、确认，还是自动触发
- 动作后是否需要前置校验、二次确认或延迟处理

4. 条件与规则
- 在什么条件下可以继续
- 在什么条件下会分支
- 有哪些资格判断、限制条件、开关、阈值、互斥关系、排序规则

5. 结果与反馈
- 成功后页面如何变化
- 失败后页面如何变化
- 是否有 toast、弹窗、文案、空态、禁用态、局部刷新、跳转、返回
- 用户最终知道自己“成功了 / 失败了 / 部分成功了”的依据是什么

6. 风险与异常
- 最容易出错的地方是什么
- 最容易引发投诉、误操作、数据错误、重复提交、状态冲突的地方是什么
- 失败后是否支持重试、撤销、回退、人工处理、fallback

##### 二、建议拆解顺序

在真正写 bullet 前，建议按以下顺序组织思路：

1. 先写默认展示与入口
2. 再写主流程
3. 再写条件分支
4. 再写状态切换
5. 再写异常和失败处理
6. 最后补边界、限制和高风险误操作反馈

也就是说，不能一上来就写一堆边界情况；  
也不能只写 happy path 就结束。

##### 三、四类思考工具（必须执行）

###### 1. EP（等价类）

目的：
- 先识别哪些用户、对象、输入、状态会被“同一套逻辑”处理，哪些会有差异

要问：
- 不同角色是否有不同入口、不同权限、不同展示
- 不同对象类型是否处理逻辑不同
- 不同输入类型是否会走不同路径
- 不同状态下页面是否会展示不同内容

如果存在差异，就必须写进交互描述里，不能默认“一套逻辑通用”。

###### 2. BVA（边界值）

目的：
- 找到那些最容易被忽略、但最容易出 bug 的临界条件

要问：
- 是否存在最小值、最大值、上限、下限
- 是否存在时间窗口
- 是否存在数量限制、次数限制、长度限制
- 状态切换是否存在临界点

如果存在边界，就必须在交互描述里体现：
- 到边界前怎样
- 到边界后怎样
- 超过边界如何提示

###### 3. ST（场景法）

目的：
- 保证写的不是单一流程，而是完整场景覆盖

至少要覆盖：
- 主流程
- 备选流程
- 异常流程

要问：
- 正常情况下用户怎么完成任务
- 如果路径切换了，是否有备选流程
- 如果系统失败、条件不满足、用户中断，会怎样

如果只写主流程，不算完成。

###### 4. EG（错误推测）

目的：
- 主动寻找最可能被骂、最可能引发投诉、最可能导致损失的交互点

要问：
- 用户最可能在哪一步误操作
- 哪一步最容易重复提交
- 哪一步最容易造成状态错乱
- 哪一步最容易“用户以为成功，其实失败”
- 哪一步失败后如果提示不清，会造成什么问题

这些不是“可选补充”，而是高风险需求必须写进去。

##### 四、必须覆盖的交互元素

如果某个需求模块存在以下内容，不得省略：

1. 登录态 / 未登录态差异
2. 有权限 / 无权限差异
3. 空态 / 满态 / 无数据态
4. 成功 / 失败 / 部分失败结果
5. 关键阈值边界
6. 返回、取消、关闭、重试、撤销、回退逻辑
7. 高风险误操作后的系统反馈
8. 关键文案和提示语
9. 重要字段、格式规则、展示规则
10. 与上下一个状态或上下一个页面的衔接关系

##### 五、写作粒度要求

用户交互详细描述必须细到以下程度：

- 能看出“用户先看到什么”
- 能看出“用户接下来做什么”
- 能看出“系统在什么条件下如何判断”
- 能看出“页面会进入什么状态”
- 能看出“成功和失败分别长什么样”
- 能看出“最容易出问题的地方如何处理”

如果写完以后，测试仍然不知道怎么测，设计仍然不知道怎么画，研发仍然不知道什么时候分支，那说明粒度还不够。

##### 六、不合格写法

以下写法都视为不合格：

- 只写一句“点击后进入下一步”
- 只写 happy path，不写分支
- 只写“提示错误”，不写错误怎么提示
- 只写“展示列表”，不写空态/无数据/格式规则
- 只写“提交成功”，不写成功后页面怎么变化
- 只写“失败后重试”，不写用户怎么知道可以重试
- 只写结果，不写条件和状态变化

### 6.3 实现职责与依赖方评估（与需求列表平级）

这一部分不跟着某一个大模块写，而是跟整个需求平级输出。  
目的不是只拆“前端 / 后端 / 算法”，而是先判断：这个需求实际会牵涉哪些实现角色和依赖方，再分别写清楚各自的职责边界。

#### 判断顺序

先不要直接写职责，而是先逐个判断哪些角色与本需求相关。  
如果相关，就写职责；如果不相关，也要明确写 `不涉及`，避免遗漏。

#### 角色相关性判断标准

| 角色 | 什么时候算相关 | 写法要求 |
|------|----------------|----------|
| 前端 | 只要涉及任何一端的体验、页面、交互、状态展示、反馈设计，就相关；角色可以是用户、运营、商家任意一端 | 写该角色对应页面、状态、交互承接，不写按钮级动作 |
| 后端 | 只要涉及逻辑处理、状态回传、数据读写、接口编排、任务流转、记录落库、权限判断、规则执行，就相关 | 写服务能力、接口能力、状态管理、落库与编排职责 |
| 算法 | 只要涉及推荐、召回、排序、RAG、意图识别、生成、预测、模型判断、策略计算，就相关 | 写模型/算法承担的能力边界，不写实现细节代码 |
| 测试 | 只要需求存在主流程、异常流程、边界规则、系统联调、回归影响，就相关 | 写测试重点、联调重点、回归范围 |
| DA / 数据分析 | 只要需求要看结果、看转化、看留存、看效果归因、看实验结果，就相关 | 写指标设计、归因验证、分析支持职责 |
| 设计 | 只要涉及页面、交互、组件、状态、文案、视觉承载，就相关 | 写设计产出范围和关键页面/状态设计职责 |
| 风控 | 只要涉及金额、算账、退款、发放、资格判断、越权、刷取、滥用风险，就相关 | 写资格校验、风险规则、风控策略介入点 |
| 法务 | 只要涉及合规声明、用户协议、营销条款、资格文案、敏感场景承诺，就相关 | 写法务审阅点和约束边界 |
| 文案 / 内容 | 只要涉及用户提示语、营销文案、AI 回复模板、拒答文案、后台说明文案，就相关 | 写文案资产和内容维护职责 |
| 用户端产品 | 只要需求落在 C 端或终端用户体验，就相关 | 写用户端范围、依赖和边界 |
| 商家端产品 | 只要需求涉及商家操作台、商家配置、商家营销、商家经营动作，就相关 | 写商家端范围、依赖和边界 |
| 运营端产品 | 只要需求涉及后台管理、活动运营、配置维护、人工处理、数据查看，就相关 | 写运营后台范围、依赖和边界 |
| 外部 API / 第三方服务 | 只要依赖第三方支付、消息、库存、数据、登录、AI 服务等外部能力，就相关 | 写第三方依赖和失败风险 |
| 数仓 | 只要需求要沉淀指标、事件、标签、实验数据、归因分析，就相关 | 写数仓同步、事件入仓、分析数据准备 |
| 中台 | 只要需求复用优惠券、会员、订单、消息、商品、用户等中台能力，就相关 | 写中台复用能力和改造边界 |

#### 输出格式

固定格式：

| 角色 | 是否相关 | 职责描述 |
|------|----------|----------|

写法要求：
- `是否相关` 必须明确写 `是 / 否`
- 若为 `是`，写总结型职责
- 若为 `否`，职责描述写 `不涉及`
- 职责必须是能力级、模块级、系统级描述，不写按钮级动作

### 6.4 要通过excalidraw来写用户流程图

要像用户主动询问是否要补充一个用户流程图，如果用户说：”需要“，则忘在；如果一个需求同时涉及多角色、多部门、多系统协同，AI 必须主动告知用户：当前需求适合补一张“系统框架图 / 角色关系图”，用于说明：
- 哪些部门 / 角色 / 系统参与
- 哪些是本需求范围
- 哪些是上下游依赖
- 哪些只是提供基础能力

此时应主动说明调用原因，例如：
- “当前需求涉及销售、主管、运营、商家和优惠券能力等多个角色与系统，为了帮助厘清上下游关系，建议补一张系统框架图。”

调用 `excalidraw-diagram-skill` 生成概念关系图或系统框架图。
如果用户不需要，也要明确说明：本轮不生成该图，原因是当前信息已经足够通过文字表达，或当前需求复杂度不需要额外补图。

对应的提示词用这个：
生成一张“角色关系图 / 范围与依赖”图（不是系统交互图、不是时序图；不画RPC/接口调用链路）。

【目标】
- 10秒看懂：参与角色/部门、责任边界、协作关系、范围内交付、上下游依赖。

【版式（固定）】
- 左侧：角色/部门区（分两组）
  1) 主链路角色（按业务推进顺序自上而下排列）
  2) 治理/支持角色（并列一组：审批/预算/风控/数据/运营/客服等，按需出现）
- 右侧：能力区（分两层容器）
  1) 本需求范围（新增/改造能力）= 实线边框
  2) 上下游依赖（复用/不改造）= 虚线边框
- 右下角可选：关键政策/约束（3–6条，作为“证据卡片”，不写接口参数）

【连线规则（强约束）】
- 箭头只表达“责任/动作/协作”，每条箭头必须带动词标签（例如：发起、审批、配置、兑换、发放、领取、核销、对账、冻结、复盘等）。
- 主链路动作 与 治理/护栏动作 用不同颜色/样式区分（让结构一眼可区分）。
- 控制复杂度：总箭头 8–12 条；避免把图画成系统调用链路。
- 系统/能力块只作为“承载/依赖/被治理”的对象出现；不要用它们串成调用链。

【范围表达（强约束）】
- 每个能力块必须标注：`范围内(新增/改造)` 或 `依赖(复用)`。

【输入（每次只改这里）】
- 图标题：{title}
- 主链路角色（按顺序）：
  - {actor_1}: {responsibility_1}
  - {actor_2}: {responsibility_2}
  - {actor_3}: {responsibility_3}
  - （可增减）
- 治理/支持角色（按需）：
  - {govern_role_1}: {govern_resp_1}
  - {govern_role_2}: {govern_resp_2}
  - （可增减）
- 本需求范围（新增/改造能力，3–7项）：
  - {in_scope_1}
  - {in_scope_2}
  - ...
- 上下游依赖（复用/不改造，2–5项）：
  - {dep_1}
  - {dep_2}
  - ...
- 关键政策/约束（可选，3–6条）：
  - {constraint_1}
  - {constraint_2}
  - ...

【输出】
- 输出 `.excalidraw` JSON（画面留白充足、文字不拥挤、标签可读）。
- 通过“去文字仍能看懂结构”的同构测试：主链路与治理链路的结构必须明显不同。



### 6.5 不合格写法

以下写法都不合格：
- 一个大模块只写一整段话
- 页面描述列里塞满交互细节
- 用户交互详细描述没有分层 bullet
- 一个小模块没有条件分支和状态差异
- 模块细到按钮、控件、toast
- 没明确这是用户端、商家端还是运营后台端能力
- 展示内容、文案、格式规则、失败提示没有写清
