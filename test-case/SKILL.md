# QA 测试用例技能

> 本文件用途：定义测试用例技能的主流程、输入要求、输出模板和三种工作模式（生成 / 补全 / 评审）。

当用户要求生成、补全或评审测试用例时，根据 PRD 文档、页面流程、接口说明、交互原型、截图或已有 case 草稿进行学习，并按本文件模板与规范输出。

## 执行前

- **输入范围**：用户提供的 PRD/需求摘要、页面流程、接口说明、交互原型、已有测试用例草稿，或指定的功能模块。
- 若用户只说「写测试用例」「生成 case」等未提供任何上下文，必须先询问：
  - 功能/模块名称
  - 对应需求 ID
  - 测试目标（功能 / 异常 / 边界 / 回归）
  - 输出格式（表格或列表）
- 只有确认至少具备 **功能主题 + 需求或流程信息** 后，才进入正式输出。
- 缺项统一使用 `[请补充：xxx]` 或 `[待确认]` 占位，不要自行脑补。

## 输出模式

根据用户意图，选择以下模式之一：

1. **生成模式**：从需求、流程、PRD、截图直接生成一组测试用例。
2. **补全模式**：对已有用例补充缺失字段、异常场景、边界 case 或追溯信息。
3. **评审模式**：检查已有用例是否完整、可执行、可验证、可追溯，并输出必改 / 建议 / 可选清单。

## 执行流程（文字版）

1. 先判断用户是要“写 / 补全 / 评审”测试用例。
2. 再判断当前上下文是否足够。
   - 如果上下文不够，先追问功能模块、需求 ID、流程和输出格式。
3. 当上下文足够后，识别当前属于：
   - 生成模式
   - 补全模式
   - 评审模式
4. 如果是生成模式或补全模式：
   - 优先按本文件结构直接输出。
   - 如需更完整模板、命名规范、覆盖维度，再读取 `reference.md`。
5. 如果是评审模式：
   - 先检查字段完整性。
   - 如用户提供了 Markdown 用例文件且需要字段检查，可执行 `scripts/check-testcase-fields.sh`。
   - 再输出必改 / 建议 / 可选清单。
6. 最终默认输出 Markdown 测试用例，优先使用表格；当步骤较长时可改为列表。

## 用例结构（必填字段）

每条用例至少包含以下字段：

1. **用例 ID**：如 `TC-001`，若用户未指定编号规则，可用 `[待编号]`。
2. **标题**：一句话说明验证场景，建议格式为「模块 - 场景 - 预期」。
3. **关联需求**：需求 ID、用户故事 ID、接口或页面标识；无则写 `[请补充需求来源]`。
4. **前置条件**：账号状态、环境、数据、开关或依赖准备；无则写「无」。
5. **测试步骤**：按顺序编号，每步只写一个动作或一个关键验证点。
6. **预期结果**：与步骤或整条用例对应，必须可观察、可判断通过/失败。
7. **优先级**：P0 / P1 / P2 或团队约定级别；未指定则根据主路径 / 高风险场景判断。

## 用例结构（可选字段）

- **测试类型**：冒烟 / 功能 / 回归 / 异常 / 边界 / 兼容性 / 权限。
- **测试数据**：明确输入数据、样本或账号。
- **备注**：依赖限制、暂不覆盖说明、相关缺陷单。
- **执行结果字段**：结果、执行人、执行时间、缺陷单号（若团队把设计与执行放在同一模板中）。

## 用例书写规范

- **标题**：避免笼统写法（如「登录测试」），改为具体场景（如「登录 - 输入正确账号密码后成功进入首页」）。
- **步骤**：必须可执行，避免「进行登录操作」这类抽象描述，改为「输入已注册邮箱与正确密码，点击登录按钮」。
- **预期结果**：必须可验证，避免「功能正常」「结果正确」，改为「跳转至首页，页面右上角显示用户名，接口返回 200 且 body 含 token」。
- **覆盖分类**：默认优先覆盖主路径、异常路径、边界条件；若用户只要求核心 case，可先生成 P0 / P1。
- **关联需求**：每条用例至少关联一个需求 ID 或来源描述，避免孤立 case。
- **避免重复**：多个 case 仅在输入或前置条件不同的，应明确区分差异，不要只换标题。

## 输出格式

- 默认输出为 **Markdown 表格**；若步骤较长或每步都需写预期，可改为 **分条列表**。
- 若生成的是一组用例，先给出 **覆盖范围摘要**（覆盖了哪些主路径 / 异常 / 边界），再给出 case 明细。
- 占位符统一为 `[请补充：xxx]` 或 `[待确认]`。
- 完整模板、命名规范与覆盖维度见 `reference.md`。

## 示例输出（节选）

```markdown
## 覆盖范围摘要
- 核心登录成功路径
- 密码错误异常路径
- 未注册邮箱异常路径
- 密码为空边界场景

| 用例 ID | 标题 | 关联需求 | 前置条件 | 步骤 | 预期结果 | 优先级 | 类型 |
|---------|------|----------|----------|------|----------|--------|------|
| TC-001 | 登录 - 正确账号密码登录成功 | R1 | 用户已注册且账号状态正常 | 1. 打开登录页 2. 输入已注册邮箱 3. 输入正确密码 4. 点击登录 | 跳转至首页，显示登录态，接口返回 200 且含 token | P0 | 功能 |
| TC-002 | 登录 - 密码错误时提示错误信息 | R1 | 用户已注册且账号状态正常 | 1. 打开登录页 2. 输入已注册邮箱 3. 输入错误密码 4. 点击登录 | 页面提示“账号或密码错误”，停留在登录页，不创建登录态 | P0 | 异常 |
| TC-003 | 登录 - 密码为空时按钮置灰或提示必填 | R1 | 无 | 1. 打开登录页 2. 输入已注册邮箱 3. 密码留空 4. 点击登录按钮 | 按钮不可点击或提示“密码不能为空”，不发起登录请求 | P1 | 边界 |
```

若用户提供的是 **已有测试用例** 并要求“评审”“检查是否规范”，则输出：符合项 ✓、缺失字段、不可执行步骤、不可验证预期、未覆盖的关键场景；按 **必改 / 建议 / 可选** 分级，并给出修改建议。
